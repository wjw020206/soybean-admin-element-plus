# 第 10 课：登录流程源码拆解

## 课程定位

第 9 课我们已经理解了 Pinia store 的整体结构，知道了 `auth store` 是认证状态中心。本课继续沿着这条主线往下走，专门拆开“用户名密码登录”这条流程。

很多初学者看后台管理项目时，会把登录理解成“点一下按钮，接口成功后跳转首页”。这太粗了。真实项目里的登录流程至少包含这些环节：

```txt
输入账号密码
  -> 表单校验
  -> 调登录接口
  -> 获取 token / refreshToken
  -> 写入本地存储
  -> 再请求用户信息
  -> 更新 auth store
  -> 判断是否需要清空旧 tabs
  -> 跳转到目标页或首页
  -> 弹出登录成功提示
```

本课就是要把这条链路拆细。你要知道每一步在哪个文件里发生、为什么这么安排、如果某一步失败了会怎么处理。

## 学习目标

学完本课后，你应该能够：

- 说明密码登录页面是如何提交表单的。
- 理解登录页为什么不直接调用 `fetchLogin`，而是调用 `authStore.login()`。
- 理解 `fetchLogin`、`fetchGetUserInfo` 分别负责什么。
- 理解 token / refreshToken 为什么先写入本地存储，再拉取用户信息。
- 理解 `loginByToken()` 在登录流程中的作用。
- 理解 `initUserInfo()` 的作用和使用场景。
- 理解登录成功后为什么要判断是否清空 tabs。
- 理解 `redirectFromLogin()` 如何处理登录后重定向。
- 理解登录失败后的清理逻辑。
- 能够画出“点击登录按钮到进入首页”的完整流程图。

## 四、为什么登录页不直接调 `fetchLogin`

`fetchLogin()` 只是接口函数，它只知道：

```txt
拿着账号密码去请求后端
```

但登录成功后还有很多事情要做：

- 缓存 token。
- 缓存 refreshToken。
- 请求用户信息。
- 更新 `auth store`。
- 判断是否清空标签页。
- 处理登录后跳转。
- 弹出成功通知。

这些都不应该写在页面组件里。

所以项目把完整业务链路放在：

```txt
authStore.login()
```

页面只负责触发动作，不负责处理整条认证主线。这是一个非常好的后台项目分层方式。

## 五、表单校验先于登录请求

登录提交函数是：

```ts
async function handleSubmit() {
  await validate();
  await authStore.login(model.value.userName, model.value.password);
}
```

意思是：

```txt
先校验，再请求
```

为什么要先 `await validate()`？

因为账号密码登录是一个典型表单提交场景：

- 用户名不能为空。
- 密码不能为空。
- 格式可能还有限制。

如果不先校验，页面会把明显无效的数据直接发给后端。

所以完整逻辑是：

```txt
点击按钮
  -> ElForm 校验
  -> 校验通过才进入登录链路
```

## 六、`authStore.login()` 是本课的核心入口

在 [src/store/modules/auth/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/auth/index.ts) 中，登录主函数是：

```ts
async function login(userName: string, password: string, redirect = true) {
  startLoading();

  const { data: loginToken, error } = await fetchLogin(userName, password);

  if (!error) {
    const pass = await loginByToken(loginToken);

    if (pass) {
      const isClear = checkTabClear();
      let needRedirect = redirect;

      if (isClear) {
        needRedirect = false;
      }
      await redirectFromLogin(needRedirect);

      window.$notification?.success({
        title: $t('page.login.common.loginSuccess'),
        message: $t('page.login.common.welcomeBack', { userName: userInfo.userName }),
        duration: 4500
      });
    }
  } else {
    resetStore();
  }

  endLoading();
}
```

你现在不要急着逐行抠，先把这个函数拆成六步：

```txt
1. 开启 loading
2. 调登录接口 fetchLogin
3. 如果成功，执行 loginByToken
4. 如果 loginByToken 成功，处理 tabs 和跳转
5. 弹出登录成功通知
6. 如果失败，重置认证状态
7. 关闭 loading
```

这就是本项目的登录总控函数。

## 七、`loginLoading` 是怎么来的

在 `auth store` 开头有：

```ts
const { loading: loginLoading, startLoading, endLoading } = useLoading();
```

然后页面按钮绑定：

```vue
<ElButton type="primary" size="large" round block :loading="authStore.loginLoading" @click="handleSubmit">
```

所以登录按钮的 loading 状态来自 store，而不是组件自己维护。

这有什么好处？

```txt
登录动作和登录中的状态放在一起
页面组件不需要自己额外维护一个 loading ref
```

对于后台项目来说，这样更统一。

## 八、第一步：`fetchLogin()` 只做登录请求

`fetchLogin()` 在 [src/service/api/auth.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/service/api/auth.ts)：

```ts
export function fetchLogin(userName: string, password: string) {
  return request<Api.Auth.LoginToken>({
    url: '/auth/login',
    method: 'post',
    data: {
      userName,
      password
    }
  });
}
```

这个函数非常纯粹，只做三件事：

- 指定接口地址 `/auth/login`
- 指定请求方式 `post`
- 指定请求体 `{ userName, password }`

并且通过泛型声明响应类型：

```ts
Api.Auth.LoginToken
```

它对应的类型是：

```ts
interface LoginToken {
  token: string;
  refreshToken: string;
}
```

所以你可以理解为：

```txt
fetchLogin 的职责只是“拿 token”
不负责登录后的业务处理
```

## 九、为什么登录成功后还要再调一次 `getUserInfo`

很多初学者会有这个疑问：

```txt
登录接口不是已经成功了吗？
为什么还要再请求一次用户信息？
```

原因是登录接口通常返回的只是认证凭证：

```ts
{
  token: '...',
  refreshToken: '...'
}
```

但后台项目真正需要的还有：

- 当前用户 id
- 用户名
- 角色
- 按钮权限

这些信息通常由另一个接口提供：

```ts
fetchGetUserInfo()
```

也就是说：

```txt
登录接口负责“证明你是谁”
用户信息接口负责“告诉系统你有什么权限、该展示什么内容”
```

所以完整登录流程通常是两步：

```txt
登录 -> 拿 token
带 token 拉用户信息 -> 初始化权限上下文
```

## 十、`loginByToken()`：连接“拿到 token”和“完成登录”的桥梁

`loginByToken()` 是本课非常关键的中间函数：

```ts
async function loginByToken(loginToken: Api.Auth.LoginToken) {
  // 1. stored in the localStorage, the later requests need it in headers
  localStg.set('token', loginToken.token);
  localStg.set('refreshToken', loginToken.refreshToken);

  // 2. get user info
  const pass = await getUserInfo();

  if (pass) {
    token.value = loginToken.token;

    return true;
  }

  return false;
}
```

这段逻辑一定要看懂。它表达的是：

```txt
登录接口成功 != 登录流程完成
只有 token 保存成功、用户信息拉取成功后，才算真正完成登录
```

这里顺序不能乱：

### 第一步：先把 token / refreshToken 写入本地存储

```ts
localStg.set('token', loginToken.token);
localStg.set('refreshToken', loginToken.refreshToken);
```

为什么必须先写？

因为后面的 `fetchGetUserInfo()` 本质上还是一个受保护接口，请求层会从本地存储里读取 token，自动放进请求头。

如果你还没存 token，就去请求用户信息，很可能拿不到认证信息。

所以顺序必须是：

```txt
先保存 token
再请求用户信息
```

### 第二步：再请求用户信息

```ts
const pass = await getUserInfo();
```

如果用户信息没拿到，就说明登录上下文没有完整初始化成功。

### 第三步：最后再更新 store 里的 `token`

```ts
token.value = loginToken.token;
```

这个 `token` 是 auth store 自己的响应式状态。写入之后，依赖它的计算属性：

```ts
const isLogin = computed(() => Boolean(token.value));
```

也就成立了。

## 十一、`getUserInfo()` 真正更新了什么

`getUserInfo()` 是：

```ts
async function getUserInfo() {
  const { data: info, error } = await fetchGetUserInfo();

  if (!error) {
    // update store
    Object.assign(userInfo, info);

    return true;
  }

  return false;
}
```

核心是：

```ts
Object.assign(userInfo, info);
```

说明用户信息不是重新替换了一个新的响应式对象，而是把接口返回值合并到现有的 `reactive(userInfo)` 上。

`userInfo` 初始值是：

```ts
const userInfo: Api.Auth.UserInfo = reactive({
  userId: '',
  userName: '',
  roles: [],
  buttons: []
});
```

接口成功后，它变成真正的用户信息。

这会直接影响后面的很多逻辑：

- `authStore.isStaticSuper`
- 路由权限过滤
- 按钮权限控制
- 顶部用户名称显示

所以 `getUserInfo()` 是认证链路里的第二个关键节点。

## 十二、登录成功后为什么还要 `checkTabClear()`

这段逻辑很多人第一次看会忽略：

```ts
const isClear = checkTabClear();
let needRedirect = redirect;

if (isClear) {
  needRedirect = false;
}
await redirectFromLogin(needRedirect);
```

而 `checkTabClear()` 的逻辑是：

```ts
function checkTabClear(): boolean {
  if (!userInfo.userId) {
    return false;
  }

  const lastLoginUserId = localStg.get('lastLoginUserId');

  if (lastLoginUserId !== userInfo.userId) {
    localStg.remove('globalTabs');
    tabStore.clearTabs();

    return true;
  }

  return false;
}
```

它解决的问题是：

```txt
如果这次登录的用户和上次登录的用户不是同一个人，
就不应该继续保留上一个用户留下来的 tabs。
```

为什么？

因为后台系统里的 tabs 往往带着用户上下文：

- 打开的页面不同。
- 权限不同。
- 标签标题和缓存状态不同。

如果用户 A 退出后，用户 B 登录，还保留 A 的 tabs，体验和权限都可能出问题。

所以这里做了一层“用户切换时的 tab 清理”。

## 十三、`lastLoginUserId` 是什么时候记录的

登录流程里不是一开始就写 `lastLoginUserId`，而是在重置时记录：

```ts
function recordUserId() {
  if (!userInfo.userId) {
    return;
  }

  localStg.set('lastLoginUserId', userInfo.userId);
}
```

它在 `resetStore()` 中调用：

```ts
async function resetStore() {
  recordUserId();

  clearAuthStorage();

  authStore.$reset();

  if (!route.meta.constant) {
    await toLogin();
  }

  tabStore.cacheTabs();
  routeStore.resetStore();
}
```

意思是：

```txt
退出登录或认证失效前，
先把当前用户 id 记下来，
下次登录时再和新用户比较。
```

这是一种很典型的后台项目细节处理。

## 十四、登录成功后的跳转：`redirectFromLogin()`

路由跳转逻辑在 [src/hooks/common/router.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/hooks/common/router.ts)：

```ts
async function redirectFromLogin(needRedirect = true) {
  const redirect = route.value.query?.redirect as string;

  if (needRedirect && redirect) {
    await routerPush(redirect);
  } else {
    await toHome();
  }
}
```

这段逻辑的含义是：

```txt
如果登录前是因为访问受保护页面被拦截到登录页，
并且当前允许重定向，
那登录后就回到原来要去的页面；
否则跳首页。
```

### 为什么登录页会有 `redirect`

因为路由守卫里会把目标地址带到登录页：

```ts
if (!isLogin) {
  return { name: loginRoute, query: { redirect: to.fullPath } };
}
```

所以如果用户原本想访问：

```txt
/manage/user
```

但没登录，就会先被带到：

```txt
/login/pwd-login?redirect=/manage/user
```

登录成功后再回去，这是很常见的后台管理体验。

## 十五、登录失败后的清理：`resetStore()`

在登录逻辑里：

```ts
if (!error) {
  ...
} else {
  resetStore();
}
```

说明只要登录请求失败，就会走认证重置。

`resetStore()` 的作用不只是清 token，而是做整套清理：

```txt
记录上次用户 id
清 token / refreshToken
重置 auth store
必要时跳登录页
缓存当前 tabs
重置 route store
```

你要理解：

```txt
登录失败 / 认证失效
不是一个“提示报错”就完了的事情，
它意味着整套认证上下文可能失效，需要恢复到未登录状态。
```

## 十六、`clearAuthStorage()` 只做一件事

在 [src/store/modules/auth/shared.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/auth/shared.ts) 中：

```ts
export function clearAuthStorage() {
  localStg.remove('token');
  localStg.remove('refreshToken');
}
```

这个函数很短，但职责很清晰：

```txt
只清认证相关存储
不做跳转
不做 store 重置
不做界面提示
```

为什么要拆出来？

因为这类底层清理操作往往会在多个场景复用：

- 登录失败
- 主动退出登录
- token 刷新失败

把底层存储清理抽成一个小函数，是非常合理的封装。

## 十七、`initUserInfo()` 是给什么场景准备的

`auth store` 里还有：

```ts
async function initUserInfo() {
  const hasToken = getToken();

  if (hasToken) {
    const pass = await getUserInfo();

    if (!pass) {
      resetStore();
    }
  }
}
```

它不是登录按钮点下去时调用的主入口，而是：

```txt
页面刷新后，浏览器里还有 token，
但内存中的 auth store 已经重新初始化了，
这时要重新把 userInfo 拉回来。
```

所以它解决的是：

```txt
刷新页面后的认证恢复
```

后面学习路由守卫时，你会看到 route 初始化会依赖它。

## 十八、为什么登录成功通知放在 store 而不是组件里

登录成功后：

```ts
window.$notification?.success({
  title: $t('page.login.common.loginSuccess'),
  message: $t('page.login.common.welcomeBack', { userName: userInfo.userName }),
  duration: 4500
});
```

这段通知写在 `authStore.login()` 里，而不是 `pwd-login.vue` 里。

原因是：

```txt
“登录成功”是认证业务结果，不是登录页面局部行为
```

以后如果换成：

- 快捷账号登录
- 手机验证码登录
- 某些单点登录回调

只要它们最终都走 `authStore.login()` 或类似认证完成逻辑，这个提示都可以复用。

这就是把业务反馈写在 store 的价值。

## 十九、完整登录链路总图

把前面内容串起来，完整链路可以写成：

```txt
PwdLogin.handleSubmit()
  -> validate()
  -> authStore.login(userName, password)
    -> startLoading()
    -> fetchLogin(userName, password)
      -> POST /auth/login
      -> 返回 { token, refreshToken }
    -> loginByToken(loginToken)
      -> localStg.set('token', token)
      -> localStg.set('refreshToken', refreshToken)
      -> getUserInfo()
        -> fetchGetUserInfo()
        -> Object.assign(userInfo, info)
      -> token.value = loginToken.token
    -> checkTabClear()
      -> 对比 lastLoginUserId 和当前 userId
      -> 必要时清理 tabs
    -> redirectFromLogin(needRedirect)
      -> 有 redirect 且允许跳转：回原页面
      -> 否则跳首页
    -> notification.success(...)
    -> endLoading()
```

如果你能自己默写出这条链路，本课就算真正掌握了。

## 二十、什么时候算“登录成功”

这个问题你最好单独想一遍。

在这个项目里，“登录成功”不是指：

```txt
/auth/login 接口返回成功
```

而是指：

```txt
登录接口成功
并且 token / refreshToken 已缓存
并且用户信息已成功拉取
并且 auth store 已进入登录态
```

这点非常重要。

因为在后台管理系统里，只有 token 没有用户信息，很多页面其实还是没法正常工作的。

## 二十一、阅读这节课时容易掉进的误区

### 误区 1：把接口函数当成业务入口

不是：

```ts
fetchLogin()
```

而是：

```ts
authStore.login()
```

### 误区 2：认为 token 存下来就结束了

不是。

后面还有：

- 获取用户信息
- 初始化权限上下文
- 登录后跳转

### 误区 3：忽略失败后的清理

真实项目里，失败后的恢复动作和成功路径一样重要。

### 误区 4：只盯着登录页组件

登录页组件只是触发点，核心逻辑在 store 和 service 层。
