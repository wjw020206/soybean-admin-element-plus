# 第 12 课：请求封装与错误处理

## 课程定位

第 10 课我们理解了登录流程，第 11 课理解了 token、refreshToken 等认证状态存在哪里。接下来要继续往下走一个关键问题：

```txt
这些 token 到底是怎么参与每一次请求的？
为什么业务代码里写 request({ url, method }) 就够了？
后端 code 不同的时候，项目为什么会自动登出、弹窗或重试请求？
```

这就是本课要讲的请求层。

后台管理项目里，请求层不是简单地“axios 发请求”。真实项目通常会统一处理：

- baseURL
- 代理环境
- Authorization
- 响应数据结构转换
- 后端 success code 判断
- token 过期
- 登出逻辑
- 错误提示去重
- 多服务地址

如果这些逻辑散落在页面里，项目很快会变得混乱。所以这个仓库把它们统一收拢到了：

```txt
src/service/request
packages/axios
src/utils/service.ts
```

本课的目标，就是把这套请求封装看懂。

## 学习目标

学完本课后，你应该能够：

- 理解本项目为什么要封装请求层。
- 区分 `createRequest` 和 `createFlatRequest`。
- 理解 `request` 实例是如何创建出来的。
- 理解 `baseURL` 和 `otherBaseURL` 是如何根据环境变量生成的。
- 理解请求前如何自动添加 `Authorization`。
- 理解后端响应为什么要经过 `transform`。
- 理解 `isBackendSuccess` 如何判断“业务成功”。
- 理解 `onBackendFail` 和 `onError` 的职责区别。
- 理解为什么不同错误码会触发不同处理逻辑。
- 理解 `demoRequest` 为什么存在，以及它和主请求实例有什么不同。

## 二、为什么后台项目一定会封装请求层

如果没有封装，页面代码可能会变成这样：

```ts
axios.get('/user', {
  headers: {
    Authorization: `Bearer ${localStorage.getItem('token')}`
  }
});
```

然后另一个页面又写一遍：

```ts
axios.post('/role', data, {
  headers: {
    Authorization: `Bearer ${localStorage.getItem('token')}`
  }
});
```

这样很快会出现问题：

- token 注入逻辑重复。
- 错误提示方式不统一。
- 后端 `code` 判断到处散落。
- token 过期时很难统一刷新。
- 多个 baseURL 时很难维护。

所以项目会把这些共性问题统一抽到请求层：

```txt
页面只关心“我要请求什么”
请求层负责“怎么发、怎么判成功、出错怎么处理”
```

## 三、本项目请求层的整体结构

你可以把它理解成三层。

### 第 1 层：业务接口函数

例如：

- `fetchLogin`
- `fetchGetUserInfo`
- `fetchGetUserList`

它们位于：

```txt
src/service/api/*
```

职责是：

```txt
描述请求哪个接口、用什么方法、传什么参数
```

### 第 2 层：项目请求实例

位于：

```txt
src/service/request/index.ts
```

这里会创建：

- `request`
- `demoRequest`

职责是：

```txt
把认证、错误处理、后端 code 规则、baseURL 等项目规则配置进去
```

### 第 3 层：通用 axios 工具库

位于：

```txt
packages/axios
```

职责是：

```txt
提供 createRequest / createFlatRequest 这样的可复用底层能力
```

所以调用链是：

```txt
页面 / store
  -> src/service/api/*
  -> src/service/request/index.ts
  -> packages/axios
  -> axios
```

```ts
      return response.data.data;

    async onRequest(config) {
      ...
    },
    isBackendSuccess(response) {
      ...
    },
    async onBackendFail(response, instance) {
      ...
    },
    onError(error) {
```

你可以把它理解成：


也就是说，这里不是每次请求都重新 new 一个 axios，而是先创建一个统一请求实例，后面所有业务接口都复用它。

## 五、`createFlatRequest` 和 `createRequest` 的区别

这是本课最重要的概念之一。

在 [packages/axios/src/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/packages/axios/src/index.ts) 里，工具库提供了两个创建函数：

```ts
createRequest(...)
createFlatRequest(...)
```

### `createRequest`

返回值是：

```txt
请求成功 -> 直接返回数据
请求失败 -> 抛异常
```

更像传统 axios 风格。

### `createFlatRequest`

返回值是：

```txt
{ data, error, response }
```

也就是成功和失败都被拍平成一个统一结构。

在源码里：

```ts
return { data, error: null, response };
```

失败时：

```ts
return { data: null, error, response: error.response };
```

所以业务层可以这样写：

```ts
const { data, error } = await fetchLogin(userName, password);

if (!error) {
  ...
}
```

这就是为什么这个项目大量出现：

```ts
const { data, error } = await ...
```

而不是：

```ts
try {
  const data = await ...
} catch (error) {
  ...
}
```

## 六、为什么这个项目更偏向 `createFlatRequest`

后台项目里很多请求都要统一走：

```txt
if (!error) { ... } else { ... }
```

这种扁平结构有几个好处：

### 1. 页面和 store 写法统一

例如：

```ts
const { data: loginToken, error } = await fetchLogin(userName, password);
```

比到处写 `try/catch` 更统一。

### 2. 更适合“请求失败不一定是异常流程”的业务场景

比如：

- 登录失败
- token 过期
- 后端返回业务错误码

这些在后台项目里很常见，不一定想用异常中断业务逻辑。

### 3. 更容易和统一错误处理配合

请求层已经做了很多处理，业务层只需要判断 `error` 是否存在。

## 七、`packages/axios` 里真正的公共核心：`createCommonRequest`

在 [packages/axios/src/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/packages/axios/src/index.ts) 中：

```ts
function createCommonRequest(...) {
  const opts = createDefaultOptions(options);
  const axiosConf = createAxiosConfig(axiosConfig);
  const instance = axios.create(axiosConf);

  instance.interceptors.request.use(...)
  instance.interceptors.response.use(...)

  return {
    instance,
    opts,
    cancelAllRequest
  };
}
```

这说明：

```txt
createRequest 和 createFlatRequest 的差异主要在“最后返回值形式”
而真正的 axios 实例、拦截器、配置处理逻辑是共用的
```

这是一个很好的抽象方式。

## 八、`createAxiosConfig` 默认做了什么

在 [packages/axios/src/options.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/packages/axios/src/options.ts) 中：

```ts
const axiosConfig: CreateAxiosDefaults = {
  timeout: TEN_SECONDS,
  headers: {
    'Content-Type': 'application/json'
  },
  validateStatus: isHttpSuccess,
  paramsSerializer: params => {
    return stringify(params);
  }
};
```

这表示项目的请求层默认具备：

- 10 秒超时
- 默认 JSON 请求头
- 只把 2xx 和 304 当作 HTTP 成功
- 用 `qs.stringify` 处理 query 参数

其中：

```ts
validateStatus: isHttpSuccess
```

对应：

```ts
status >= 200 && status < 300 || status === 304
```

这说明请求层先分两层判断：

### HTTP 层成功

例如：

```txt
200
201
204
304
```

### 业务层成功

例如后端 body 里：

```txt
code === '0000'
```

这两层不能混为一谈。

## 十、多 baseURL 是怎么支持的

`App.Service` 类型里定义了：

```ts
type OtherBaseURLKey = 'demo';
```

说明这个项目除了主服务，还预留了一个：

```txt
demo 服务
```

因此在请求层里既有：

```ts
export const request = createFlatRequest(...)
```

也有：

```ts
export const demoRequest = createRequest(
  {

```

这说明：

```txt
主系统接口走 request
其他服务接口可以单独创建自己的 request 实例
```

这样比在每个业务函数里手写完整 URL 更整洁，也更容易维护。

## 十一、`transform`：为什么要把后端响应再转换一次

主请求实例里定义了：

```ts
transform(response: AxiosResponse<App.Service.Response<any>>) {
  return response.data.data;
}
```

先看项目里的后端响应类型：

```ts
type Response<T = unknown> = {
  code: string;
  msg: string;
  data: T;
};
```

也就是说，后端原始返回通常长这样：

```ts
{
  code: '0000',
  msg: 'ok',
  data: {
    token: 'xxx',
    refreshToken: 'yyy'
  }
}
```

但业务层通常不想每次都写：

```ts
response.data.data
```

所以 `transform` 的作用就是：

```txt
把后端响应壳剥掉，只把真正业务数据返回给上层
```

这也是为什么接口函数里你看到：

```ts
const { data: loginToken } = await fetchLogin(...)
```

这里的 `loginToken` 已经是：

```ts
{ token, refreshToken }
```

而不是整包后端响应。

## 十二、请求前自动加 Authorization：`onRequest`

主请求实例里：

```ts
async onRequest(config) {
  const Authorization = getAuthorization();
  Object.assign(config.headers, { Authorization });

  return config;
}
```

而 `getAuthorization()` 在 [src/service/request/shared.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/service/request/shared.ts)：

```ts
export function getAuthorization() {
  const token = localStg.get('token');
  const Authorization = token ? `Bearer ${token}` : null;

  return Authorization;
}
```

这说明：

```txt
每次通过 request 发出的请求，
在真正发出之前，都会自动尝试从 localStorage 读取 token，
并拼成 Authorization 头。
```

所以业务接口函数不需要自己再写：

```ts
headers: { Authorization: ... }
```

这就是请求层封装的价值。

## 十三、业务成功不是看 HTTP 状态码，而是看后端 code

主请求实例里：

```ts
isBackendSuccess(response) {
  return String(response.data.code) === import.meta.env.VITE_SERVICE_SUCCESS_CODE;
}
```

而 `.env` 中配置：

```env
VITE_SERVICE_SUCCESS_CODE=0000
```

这意味着：

### HTTP 成功

例如 200，只代表：

```txt
服务器成功返回了响应
```

### 业务成功

还得看：

```txt
response.data.code === '0000'
```

如果不是这个值，请求层就会认为这是“后端业务失败”，进入 `onBackendFail` 分支。

这是后台项目里非常典型的处理方式。

## 十四、`onBackendFail` 是本课最核心的逻辑

主请求实例里：

```ts
async onBackendFail(response, instance) {
  ...
}
```

它处理的不是网络失败，而是：

```txt
HTTP 成功了，
但后端业务 code 不是成功码
```

项目把这些失败又分成几类。

### 1. 直接登出类错误码

```ts
const logoutCodes = import.meta.env.VITE_SERVICE_LOGOUT_CODES?.split(',') || [];
if (logoutCodes.includes(responseCode)) {
  handleLogout();
  return null;
}
```

对应 `.env`：

```env
VITE_SERVICE_LOGOUT_CODES=8888,8889
```

含义：

```txt
收到这些 code，就直接重置认证状态并跳登录页
```

### 2. 弹窗登出类错误码

```ts
const modalLogoutCodes = import.meta.env.VITE_SERVICE_MODAL_LOGOUT_CODES?.split(',') || [];
if (modalLogoutCodes.includes(responseCode) && !request.state.errMsgStack?.includes(response.data.msg)) {
  ...
}
```

对应 `.env`：

```env
VITE_SERVICE_MODAL_LOGOUT_CODES=7777,7778
```

含义：

```txt
收到这些 code，不是立刻退出，
而是先弹出确认框告知用户，再执行登出
```

并且这里还做了一个细节处理：

```txt
错误消息去重
防止同一个弹窗重复弹出
```

### 3. token 过期类错误码

```ts
const expiredTokenCodes = import.meta.env.VITE_SERVICE_EXPIRED_TOKEN_CODES?.split(',') || [];
if (expiredTokenCodes.includes(responseCode)) {
  const success = await handleExpiredRequest(request.state);
  if (success) {
    const Authorization = getAuthorization();
    Object.assign(response.config.headers, { Authorization });

    return instance.request(response.config) as Promise<AxiosResponse>;
  }
}
```

对应 `.env`：

```env
VITE_SERVICE_EXPIRED_TOKEN_CODES=9999,9998,3333
```

含义：

```txt
如果是 token 过期，不是直接报错，
而是尝试刷新 token，
刷新成功后自动重发原请求
```

这一块第 13 课会深入讲，本课先知道整体位置和作用。

## 十五、`onError` 和 `onBackendFail` 的区别

这是一个很容易混淆的点。

### `onBackendFail`

处理：

```txt
HTTP 成功，但后端业务失败
```

例如：

- `code !== '0000'`
- token 过期
- 权限失效
- 需要登出

### `onError`

处理：

```txt
真正进入错误流程的情况
```

包括：

- 网络错误
- 超时
- 后端失败被包装成 `BACKEND_ERROR`

主请求实例里的 `onError`：

```ts
onError(error) {
  let message = error.message;
  let backendErrorCode = '';

  if (error.code === BACKEND_ERROR_CODE) {
    message = error.response?.data?.msg || message;
    backendErrorCode = String(error.response?.data?.code || '');
  }

  ...

  showErrorMsg(request.state, message);
}
```

说明它主要负责：

```txt
把最终需要展示给用户的错误提示做统一处理
```

## 十六、`BACKEND_ERROR_CODE` 是什么

在 [packages/axios/src/constant.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/packages/axios/src/constant.ts) 中：

```ts
export const BACKEND_ERROR_CODE = 'BACKEND_ERROR';
```

它不是后端返回的 code，而是请求工具层自己造的一个错误 code。

在 `packages/axios/src/index.ts` 里，当后端业务失败但没有被 `onBackendFail` 吃掉时，会创建一个 `AxiosError`：

```ts
const backendError = new AxiosError<ResponseData>(
  'the backend request error',
  BACKEND_ERROR_CODE,
  response.config,
  response.request,
  response
);
```

这样上层在 `onError` 中就能识别：

```txt
这是后端业务错误，不是纯网络错误
```

## 十七、为什么要单独做 `showErrorMsg`

在 [src/service/request/shared.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/service/request/shared.ts) 中：

```ts
export function showErrorMsg(state: RequestInstanceState, message: string) {
  if (!state.errMsgStack?.length) {
    state.errMsgStack = [];
  }

  const isExist = state.errMsgStack.includes(message);

  if (!isExist) {
    state.errMsgStack.push(message);

    window.$message?.error({
      message,
      onClose: () => {
        state.errMsgStack = state.errMsgStack.filter(msg => msg !== message);

        setTimeout(() => {
          state.errMsgStack = [];
        }, 5000);
      }
    });
  }
}
```

它解决的是：

```txt
同一个错误可能会在短时间内被多个请求同时触发，
如果不去重，页面会疯狂弹相同的错误提示。
```

所以项目用 `errMsgStack` 做了一个非常实用的去重机制。

你可以把它理解成：

```txt
“近期已经展示过的错误消息缓存”
```

## 十八、`defaultState` 是给请求实例自己存状态的

主请求实例在创建时传了：

```ts
defaultState: {
  errMsgStack: [],
  refreshTokenPromise: null
} as RequestInstanceState,
```

在 `createFlatRequest` 内部：

```ts
flatRequest.state = {
  ...opts.defaultState
} as State;
```

这说明请求实例本身也可以带状态。

这个设计很有用，因为有些请求级逻辑并不适合放到 Pinia：

- 当前是否已有一个刷新 token 的 Promise 在进行中
- 当前哪些错误消息已经弹出过

这些都更适合放在请求实例自身的 `state` 上。

## 十九、`demoRequest`：为什么要再创建一个请求实例

在 `src/service/request/index.ts` 中还有：

```ts
export const demoRequest = createRequest(
  {
    baseURL: otherBaseURL.demo
  },
  {
    transform(response: AxiosResponse<App.Service.DemoResponse>) {
      return response.data.result;
    },
    ...
  }
);
```

它和主请求实例不同的地方有两个：

### 1. baseURL 不同

它走：

```txt
demo 服务
```

### 2. 后端响应结构不同

主服务返回：

```ts
{ code, msg, data }
```

demo 服务返回：

```ts
{ status, message, result }
```

所以它的：

- `transform`
- `isBackendSuccess`
- `onError`

都需要单独配置。

这就是“多服务地址 + 多后端响应协议”时，请求层封装的价值。

## 二十、业务接口文件为什么看起来这么薄

例如 [src/service/api/auth.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/service/api/auth.ts)：

```ts

      userName,
      password
```

以及 [src/service/api/system-manage.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/service/api/system-manage.ts)：

它们之所以很薄，是因为：

```txt

```

这正是理想状态。

## 二十一、整个请求主线怎么串起来

你可以把完整请求链路总结成：

```txt
业务代码调用 fetchXxx()
  -> fetchXxx 内部调用 request({...})
  -> request 是 createFlatRequest 创建的实例
  -> 底层走 createCommonRequest -> axios 实例
  -> 请求拦截器执行 onRequest，自动加 Authorization
  -> 收到响应后先看 HTTP 是否成功
  -> 再看 isBackendSuccess 是否业务成功
  -> 成功：transform 出真正业务数据
  -> 失败：进入 onBackendFail
  -> 如果仍未解决：进入 onError
  -> 最终业务层拿到 { data, error }
```

这就是本项目请求层的主线。

## 二十二、本课最关键的工程思想

本课最重要的不是记住每一个函数名，而是理解这三个设计思想。

### 1. 请求层要统一承接“横切逻辑”

所谓横切逻辑，就是所有接口几乎都会遇到的问题：

- token
- 错误码
- 错误提示
- 登录失效
- baseURL

这些绝不应该散落在页面里。

### 2. 业务接口文件要尽量薄

理想状态下，`src/service/api/*` 应该只写：

```txt
接口地址
请求方法
参数类型
响应类型
```

其余规则都交给请求层。

### 3. HTTP 成功和业务成功是两层概念

这点非常重要。

```txt
HTTP 200 ≠ 业务成功
```

后台项目里，真正的业务成功通常还要看后端 `code` / `status`。
