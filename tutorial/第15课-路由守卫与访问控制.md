# 第 15 课：路由守卫与访问控制

## 课程定位

第 14 课我们已经把 Vue Router 初始化主线看清楚了，知道了：

```txt
createRouter(...)
  -> app.use(router)
  -> createRouterGuard(router)
  -> await router.isReady()
```

但真正决定“一个页面能不能进”的，不是 `createRouter`，而是路由守卫。

后台管理项目里，几乎所有访问控制都集中在这里：

- 未登录能不能进？
- 登录后还能不能回登录页？
- 某个页面是否需要登录？
- 当前用户有没有这个页面的角色权限？
- 当前地址落到 `not-found` 时，到底是 404、403，还是权限路由还没初始化？
- 菜单里的外链路由为什么点一下就新开标签页？

这些问题，这一课统一解决。

## 学习目标

学完本课后，你应该能够：

- 掌握本项目 `beforeEach` 的完整判断顺序。
- 理解 `initRoute(to)` 为什么放在最前面。
- 理解常量路由初始化和权限路由初始化为什么都发生在守卫里。
- 理解未登录访问、登录页重定向、登录后访问控制的处理方式。
- 理解 `to.meta.constant`、`to.meta.roles` 在权限判断中的作用。
- 理解 403 和 404 在本项目里的区分方式。
- 理解外链路由 `meta.href` 的处理方式。
- 理解进度条守卫和标题守卫在这套访问控制链路中的位置。
- 能自己画出 `beforeEach` 的判断流程图。

## 二、先看守卫入口：项目一共挂了哪些守卫

在 [src/router/guard/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/guard/index.ts)：

```ts
export function createRouterGuard(router: Router) {
  createProgressGuard(router);
  createRouteGuard(router);
  createDocumentTitleGuard(router);
}
```

也就是说，这个项目的路由守卫入口分三层：

### 1. `createProgressGuard`

负责：

```txt
路由切换前启动 NProgress
路由切换后结束 NProgress
```

对应 [src/router/guard/progress.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/guard/progress.ts)：

```ts
router.beforeEach(() => {
  window.NProgress?.start?.();
});

router.afterEach(() => {
  window.NProgress?.done?.();
});
```

### 2. `createRouteGuard`

负责：

```txt
初始化路由、登录判断、权限判断、403/404 分流、外链处理
```

这是本课主角。

### 3. `createDocumentTitleGuard`

负责：

```txt
路由切换结束后，根据 meta.title / meta.i18nKey 设置页面标题
```

对应 [src/router/guard/title.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/guard/title.ts)：

```ts
router.afterEach(to => {
  const { i18nKey, title } = to.meta;

  const documentTitle = i18nKey ? $t(i18nKey) : title;

  useTitle(documentTitle);
});
```

所以本项目的路由守卫职责分工很明确：

```txt
progress 守卫管体验
route 守卫管访问控制
title 守卫管页面标题
```

## 三、真正的核心：`beforeEach` 的总流程

在 [src/router/guard/route.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/guard/route.ts) 中：

```ts
router.beforeEach(async (to, from) => {
  const location = await initRoute(to);

  if (location) {
    return location;
  }

  const authStore = useAuthStore();

  const rootRoute: RouteKey = 'root';
  const loginRoute: RouteKey = 'login';
  const noAuthorizationRoute: RouteKey = '403';

  const isLogin = Boolean(localStg.get('token'));
  const needLogin = !to.meta.constant;
  const routeRoles = to.meta.roles || [];

  const hasRole = authStore.userInfo.roles.some(role => routeRoles.includes(role));
  const hasAuth = authStore.isStaticSuper || !routeRoles.length || hasRole;

  if (to.name === loginRoute && isLogin) {
    return { name: rootRoute };
  }

  if (!needLogin) {
    return handleRouteSwitch(to, from);
  }

  if (!isLogin) {
    return { name: loginRoute, query: { redirect: to.fullPath } };
  }

  if (!hasAuth) {
    return { name: noAuthorizationRoute };
  }

  return handleRouteSwitch(to, from);
});
```

你先不要急着记代码，先记判断顺序：

```txt
1. 先 initRoute(to)
2. 再判断是不是登录页重定向
3. 再判断这个页面需不需要登录
4. 再判断当前有没有登录
5. 再判断有没有角色权限
6. 最后处理正常切换或外链切换
```

这条顺序非常重要。

## 四、为什么 `initRoute(to)` 一定放在最前面

`beforeEach` 一开始就执行：

```ts
const location = await initRoute(to);

if (location) {
  return location;
}
```

这意味着：

```txt
后面的登录判断、权限判断，都是建立在“路由初始化已经补齐”这个前提上的。
```

为什么要这样？

因为这个项目不是一开始就把所有路由都塞进 Vue Router。

它的常量路由和权限路由都是后续初始化、动态添加的。于是第一次进入某些页面时，可能先被 `not-found` 接住。如果你不先做 `initRoute`，后面的权限判断依据就不完整。

所以 `initRoute` 的职责可以概括成：

```txt
在正式做访问控制之前，先把路由系统补到可判断状态。
```

## 五、第一层：常量路由初始化

在 `initRoute(to)` 中，先看这段：

```ts
if (!routeStore.isInitConstantRoute) {
  await routeStore.initConstantRoute();

  const path = to.fullPath;
  const location: RouteLocationRaw = {
    path,
    replace: true,
    query: to.query,
    hash: to.hash
  };

  return location;
}
```

这里的意思是：

```txt
如果常量路由还没初始化，那就先初始化；
初始化完后，重新跳回当前地址，再匹配一次。
```

为什么要“重新跳一次”？

因为用户当前进入的地址，有可能本来就是某条常量路由，但在初始化之前 Vue Router 还不知道它，于是暂时落错了。

所以处理方式不是：

```txt
初始化完就算了
```

而是：

```txt
初始化完后，拿原地址重新匹配一次
```

这就是这段 `return location` 的真正目的。

## 六、第二层：未登录时怎么处理

继续看 `initRoute(to)`：

```ts
const isLogin = Boolean(localStg.get('token'));

if (!isLogin) {
  if (to.meta.constant && !isNotFoundRoute) {
    routeStore.onRouteSwitchWhenNotLoggedIn();
    return undefined;
  }

  const loginRoute: RouteKey = 'login';
  const query = getRouteQueryOfLoginRoute(to, routeStore.routeHome);

  return {
    name: loginRoute,
    query
  };
}
```

这里先分成两种情况。

### 情况 1：未登录，但访问的是常量路由

如果：

```ts
to.meta.constant && !isNotFoundRoute
```

说明这个页面本来就是允许匿名访问的，比如：

- 登录页
- 某些异常页
- 某些公开页面

这种情况下：

```txt
允许直接访问
```

### 情况 2：未登录，且访问的不是可匿名访问页面

那就直接跳登录页：

```ts
return {
  name: loginRoute,
  query
}
```

而且这里不是简单粗暴地跳过去，还会带上 `redirect` 查询参数，为后面“登录后回跳原页面”做准备。

## 七、`redirect` 参数是怎么来的

生成登录页 query 的逻辑在：

```ts
function getRouteQueryOfLoginRoute(to: RouteLocationNormalized, routeHome: RouteKey) {
  const loginRoute: RouteKey = 'login';
  const redirect = to.fullPath;
  const [redirectPath, redirectQuery] = redirect.split('?');
  const redirectName = getRouteName(redirectPath as RoutePath);

  const isRedirectHome = routeHome === redirectName;

  const query: LocationQueryRaw = to.name !== loginRoute && !isRedirectHome ? { redirect } : {};

  if (isRedirectHome && redirectQuery) {
    query.redirect = `/?${redirectQuery}`;
  }

  return query;
}
```

你不用把每个细节都背下来，只要记住设计目的：

```txt
未登录访问受保护页面时，系统会记住你原本想去哪里，
登录成功后再决定是否回跳。
```

这就是为什么登录页 URL 经常会带一个：

```txt
?redirect=/xxx/yyy
```

## 八、第三层：已登录时先补权限路由

继续看 `initRoute(to)`：

```ts
if (!routeStore.isInitAuthRoute) {
  await routeStore.initAuthRoute();

  if (isNotFoundRoute) {
    const rootRoute: RouteKey = 'root';
    const path = to.redirectedFrom?.name === rootRoute ? '/' : to.fullPath;

    return {
      path,
      replace: true,
      query: to.query,
      hash: to.hash
    };
  }
}
```

这里的意思是：

```txt
如果用户已经登录，但权限路由还没初始化，
就先初始化权限路由。
```

这一步通常会发生在：

- 刷新页面后
- 浏览器里还有 token
- 但当前 Pinia / router 运行态刚刚重新建立

于是系统要先根据当前用户权限把 auth route 加进来。

### 为什么只在 `isNotFoundRoute` 时重定向一次

因为如果当前路由已经被 `not-found` 接住，最常见原因就是：

```txt
这条权限路由本来存在，
只是还没初始化到 Vue Router 里。
```

所以权限路由初始化完成后，要再用原路径重新匹配一次。

这也是本项目里 `not-found` 的一个关键职责：

```txt
不仅是最终 404，
也是初始化阶段的临时兜底。
```

## 九、已登录状态下为什么还要 `initUserInfo()`

在 `routeStore.initAuthRoute()` 里：

```ts
if (!authStore.userInfo.userId) {
  await authStore.initUserInfo();
}
```

而在每次已登录切路由时，还会执行：

```ts
routeStore.onRouteSwitchWhenLoggedIn();
```

里面又会：

```ts
await authStore.initUserInfo();
```

原因很简单：

```txt
localStorage 里有 token，不代表内存里的 userInfo 已经准备好了。
```

刷新页面后，token 还能从本地恢复，但 `userInfo` 是 Pinia 里的运行态，需要重新拉一次。

所以本项目把“已登录但用户信息未恢复”这个情况也纳入了守卫流程里。

## 十、第四层：登录页重定向

回到 `beforeEach` 主体：

```ts
if (to.name === loginRoute && isLogin) {
  return { name: rootRoute };
}
```

意思是：

```txt
如果当前用户已经登录，却又访问登录页，
那就把他送回首页。
```

这是后台项目里很常见的一条规则。

否则会出现一种奇怪状态：

```txt
用户明明已经在系统里，
却还能打开登录页。
```

这通常不是想要的行为。

## 十一、第五层：页面是否需要登录

接着看：

```ts
const needLogin = !to.meta.constant;

if (!needLogin) {
  return handleRouteSwitch(to, from);
}
```

这里判断页面是否需要登录，不是靠路径名，也不是靠某个白名单数组，而是靠：

```txt
route meta.constant
```

规则非常直接：

- `meta.constant === true`：常量路由，不需要登录
- 没有 `meta.constant`：默认需要登录

这就是项目访问控制的第一道业务规则。

## 十二、第六层：需要登录但未登录

接着看：

```ts
if (!isLogin) {
  return { name: loginRoute, query: { redirect: to.fullPath } };
}
```

这里和前面 `initRoute` 里的未登录处理，看起来有点像，但职责不同。

### `initRoute` 里的未登录处理

更偏：

```txt
初始化阶段的路由补齐和早期分流
```

### `beforeEach` 主体里的未登录处理

更偏：

```txt
正式访问控制判断
```

你可以理解为：

```txt
initRoute 先把系统准备好，
beforeEach 主体再做最终的访问许可判断。
```

## 十三、第七层：角色权限判断

接着看：

```ts
const routeRoles = to.meta.roles || [];

const hasRole = authStore.userInfo.roles.some(role => routeRoles.includes(role));
const hasAuth = authStore.isStaticSuper || !routeRoles.length || hasRole;
```

这里权限判断的规则是：

### 1. 如果当前用户是静态权限模式下的超级角色

```ts
authStore.isStaticSuper
```

那直接放行。

### 2. 如果当前路由没有配置 `meta.roles`

```ts
!routeRoles.length
```

也放行。

意思是：

```txt
这个页面虽然需要登录，但没有细分角色限制。
```

### 3. 如果路由配置了 `meta.roles`

那就要求：

```txt
当前用户 roles 至少命中其中一个角色
```

也就是：

```ts
authStore.userInfo.roles.some(role => routeRoles.includes(role))
```

## 十四、没有权限时为什么去 403，而不是 404

看这段：

```ts
if (!hasAuth) {
  return { name: noAuthorizationRoute };
}
```

这里跳的是：

```txt
403
```

不是 404。

因为这里的语义是：

```txt
这个页面存在，
你也已经登录，
但你没有访问它的权限。
```

这就是典型的 403。

而 404 的语义是：

```txt
这个页面不存在
```

两者不能混。

## 十五、403 和 404 在本项目里是怎么区分的

这是这套守卫里最值得你认真看的一个点。

在 `initRoute(to)` 的最后：

```ts
if (!isNotFoundRoute) {
  return undefined;
}

const exist = await routeStore.getIsAuthRouteExist(to.path as RoutePath);
const noPermissionRoute: RouteKey = '403';

if (exist) {
  return {
    name: noPermissionRoute
  };
}

return undefined;
```

这里的逻辑是：

```txt
如果当前已经被 not-found 接住，
不要立刻认定它就是 404。
先检查一下：这条路由在权限体系里到底存不存在。
```

### 如果存在

说明：

```txt
页面其实有，只是当前用户没权限
```

于是跳：

```txt
403
```

### 如果不存在

那就保留当前 `not-found` 结果，显示 404。

所以这个项目不是简单的：

```txt
没匹配上就是 404
```

而是：

```txt
先 not-found 兜底
再进一步判断这是 403 还是 404
```

## 十六、第八层：外链路由是怎么处理的

最后看：

```ts
function handleRouteSwitch(to: RouteLocationNormalized, from: RouteLocationNormalized) {
  if (to.meta.href) {
    window.open(to.meta.href, '_blank');

    return { path: from.fullPath, replace: true, query: from.query, hash: to.hash };
  }
  return undefined;
}
```

这表示：

```txt
如果当前路由配置了 meta.href，
它不是普通内部页面，而是一个外链路由。
```

处理方式不是：

```txt
在当前单页应用内部跳过去
```

而是：

```txt
直接 window.open 新开一个标签页
```

然后 router 再返回到上一个内部页面：

```ts
return { path: from.fullPath, replace: true, query: from.query, hash: to.hash };
```

这样用户在菜单里点外链时，体验就是：

```txt
当前后台页面不丢，
外部地址在新标签打开。
```

## 十七、把整个 `beforeEach` 压缩成一张脑内流程图

你现在应该把它记成下面这条链路：

```txt
进入 beforeEach
  -> 先 initRoute(to)
    -> 常量路由没初始化？先初始化，再重定向回原地址
    -> 未登录？
      -> 访问常量路由？放行
      -> 否则跳登录页，并带 redirect
    -> 已登录但权限路由没初始化？先初始化
      -> 如果当前是 not-found，重新匹配原地址
    -> 已登录且当前是 not-found？
      -> 检查路由是否存在
      -> 存在：403
      -> 不存在：404
  -> 回到 beforeEach 主体
    -> 已登录访问登录页？跳 root
    -> 页面不需要登录？继续
    -> 页面需要登录但未登录？跳登录页
    -> 没有角色权限？跳 403
    -> 正常切换
    -> 如果是外链，window.open 后退回原页
```

这就是本课最重要的结果。

```txt
路由守卫不是只做“有没有 token”这种简单判断，
```
