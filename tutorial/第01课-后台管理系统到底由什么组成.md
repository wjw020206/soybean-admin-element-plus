# 第 1 课：后台管理系统到底由什么组成

## 课程定位

这是整套课程的第一课，目标不是马上写代码，而是先建立对后台管理系统的整体认知。

很多人在学习后台模板时会直接打开 `src/views` 或复制某个页面，但这样很容易只学会“怎么改页面”，却不理解一个后台系统为什么要这样组织。后台管理系统的复杂点不只在页面，而在于登录认证、权限控制、菜单生成、接口协议、状态管理、布局体系和业务 CRUD 的协作。

本课要解决的问题是：当你看到一个后台管理项目时，应该先从哪些维度理解它。

## 学习目标

学完本课后，你应该能够：

- 说清楚后台管理系统和普通展示型网站的区别。
- 列出一个典型后台系统的核心模块。
- 理解本项目为什么适合作为后台管理系统学习样板。
- 初步知道 `SoybeanAdmin ElementPlus` 中哪些目录分别对应哪些后台能力。
- 为后续学习登录、路由、权限、请求、CRUD 和布局建立全局框架。

## 建议课时

建议用 60-90 分钟完成本课：

- 15 分钟：理解后台管理系统的业务场景。
- 20 分钟：拆解后台管理系统的核心组成。
- 20 分钟：对照本项目目录建立源码地图。
- 15 分钟：完成课后观察和功能清单。

## 一、什么是后台管理系统

后台管理系统通常不是面向普通访客的页面，而是面向企业、平台、运营、管理员或内部人员的工作台。

它的核心目标不是宣传产品，而是管理业务数据、控制系统资源、处理业务流程和提高内部工作效率。

常见后台系统包括：

- 电商后台：管理商品、订单、用户、优惠券、库存、售后。
- 内容后台：管理文章、栏目、评论、标签、作者、审核流程。
- SaaS 管理后台：管理租户、成员、角色、账单、权限、配置。
- CRM 后台：管理客户、线索、商机、跟进记录、销售人员。
- OA 后台：管理审批、员工、部门、考勤、流程。
- 运维后台：管理服务器、任务、日志、监控、告警。

这些系统虽然业务不同，但前端结构高度相似。

## 二、后台管理系统和普通网站的区别

普通网站更关注展示，后台管理系统更关注操作。

普通网站常见特点：

- 页面数量相对少。
- 权限要求较弱。
- 更多关注视觉、转化和内容展示。
- 接口交互可能较简单。
- 用户通常不需要复杂操作培训。

后台管理系统常见特点：

- 页面数量多，模块多。
- 强依赖登录和权限。
- 菜单结构复杂。
- 表格、表单、弹窗、抽屉、详情页大量出现。
- 请求接口多，且需要统一处理错误、token、分页和数据格式。
- 用户角色不同，看到的菜单和按钮可能不同。
- 需要关注操作效率、数据准确性和异常处理。

所以，后台系统不是“多写几个页面”这么简单。真正需要学习的是一套稳定的工程组织方式。

## 三、典型后台管理系统的核心模块

一个完整后台项目通常包含以下模块。

### 1. 登录认证

登录认证负责确认“你是谁”。

常见能力包括：

- 用户名密码登录。
- token 存储。
- refreshToken 刷新。
- 退出登录。
- 登录过期处理。
- 未登录访问时跳转登录页。

在本项目中，登录认证主要对应：

- `src/store/modules/auth`
- `src/service/api/auth.ts`
- `src/service/request`
- `src/utils/storage.ts`

### 2. 权限系统

权限系统负责确认“你能做什么”。

常见权限分为：

- 页面权限：能不能访问某个页面。
- 菜单权限：能不能看到某个菜单。
- 按钮权限：能不能点击新增、编辑、删除、导出等按钮。
- 数据权限：能不能看到全部数据，还是只能看到自己部门的数据。

在前端后台项目中，页面权限和菜单权限通常和路由关系最紧密。

在本项目中，权限系统主要对应：

- `src/router`
- `src/router/guard`
- `src/store/modules/route`
- `src/store/modules/auth`

### 3. 路由和菜单

后台系统通常有大量页面，路由负责页面访问，菜单负责页面导航。

一个成熟后台项目不会手写一份菜单再手写一份路由，因为这样很容易重复维护。更常见的方式是：

- 先定义路由。
- 路由中写入标题、图标、排序、权限等 meta 信息。
- 根据路由生成菜单。
- 根据当前路由生成面包屑。
- 根据路由配置决定页面是否缓存。

在本项目中，路由和菜单主要对应：

- `src/router/routes`
- `src/router/elegant`
- `src/store/modules/route`
- `src/layouts/modules/global-menu`
- `src/layouts/modules/global-breadcrumb`

### 4. 请求封装

后台系统通常接口很多，如果每个页面都自己处理 token、错误提示、后端 code、刷新 token，代码会很混乱。

所以后台项目通常会做统一请求封装：

- 统一 baseURL。
- 统一添加 Authorization。
- 统一判断后端成功 code。
- 统一处理错误提示。
- 统一处理登录过期。
- 统一转换响应数据。

在本项目中，请求封装主要对应：

- `src/service/request`
- `src/service/api`
- `src/utils/service.ts`
- `packages/axios`
- `packages/alova`

### 5. 状态管理

后台项目有很多跨页面共享的状态，例如：

- 当前用户信息。
- token。
- 当前菜单。
- 当前打开的标签页。
- 主题配置。
- 侧边栏折叠状态。
- 页面缓存列表。

这些状态不适合只放在单个组件中，因此需要 Pinia 统一管理。

在本项目中，状态管理主要对应：

- `src/store`
- `src/store/modules/auth`
- `src/store/modules/route`
- `src/store/modules/tab`
- `src/store/modules/theme`
- `src/store/modules/app`

### 6. 基础布局

后台系统通常有固定布局：

- 顶部栏。
- 侧边菜单。
- 面包屑。
- 标签页。
- 内容区。
- 页脚。
- 主题配置入口。

布局不是单纯的视觉问题，它和路由、菜单、主题、缓存、响应式都有关系。

在本项目中，布局主要对应：

- `src/layouts/base-layout`
- `src/layouts/blank-layout`
- `src/layouts/modules/global-header`
- `src/layouts/modules/global-sider`
- `src/layouts/modules/global-menu`
- `src/layouts/modules/global-tab`
- `src/layouts/modules/global-content`
- `src/layouts/modules/theme-drawer`

### 7. CRUD 页面

CRUD 是后台系统最常见的业务形态。

CRUD 代表：

- Create：新增。
- Read：查询。
- Update：编辑。
- Delete：删除。

典型后台列表页一般包含：

- 搜索表单。
- 数据表格。
- 分页。
- 新增按钮。
- 编辑按钮。
- 删除确认。
- 批量操作。
- 状态标签。
- 表单校验。
- 弹窗或抽屉。

在本项目中，最适合入门的 CRUD 示例是用户管理：

- `src/views/manage/user/index.vue`
- `src/views/manage/user/modules/user-search.vue`
- `src/views/manage/user/modules/user-operate-drawer.vue`

### 8. 主题和样式

后台系统通常需要适配企业风格，例如主色、暗色模式、菜单风格、按钮样式、表格密度等。

主题系统通常包含：

- 主题颜色。
- 暗色模式。
- Element Plus 样式覆盖。
- 全局 SCSS。
- 原子化 CSS。
- 布局尺寸配置。

在本项目中，主题和样式主要对应：

- `src/theme`
- `src/styles`
- `uno.config.ts`
- `src/store/modules/theme`

### 9. 国际化

如果后台系统需要支持多语言，就要把页面文案从组件中抽离出来。

国际化通常覆盖：

- 菜单标题。
- 按钮文案。
- 表单标签。
- 表格列名。
- 错误提示。
- 系统消息。

在本项目中，国际化主要对应：

- `src/locales`
- `src/locales/langs`
- `src/components/common/lang-switch.vue`

### 10. 工程化

后台系统通常会长期维护，因此工程化非常重要。

工程化能力包括：

- TypeScript 类型检查。
- ESLint 代码规范。
- Git commit 校验。
- 构建配置。
- 依赖管理。
- 公共包抽取。
- 路由类型生成。

在本项目中，工程化主要对应：

- `package.json`
- `tsconfig.json`
- `eslint.config.js`
- `pnpm-workspace.yaml`
- `packages`
- `build`

## 四、从本项目看后台系统的分层

可以把本项目理解为 6 层。

- `vite.config.ts`
- `tsconfig.json`

- `.env`

负责路由、状态、请求、存储、国际化等通用能力。

对应目录：

- `src/router`
- `src/store`

- `src/utils`

### 第 4 层：布局框架层

负责后台系统整体界面结构。

对应目录：

- `src/layouts`
- `src/layouts/modules`

### 第 5 层：业务页面层

负责具体业务功能。

对应目录：

- `src/views`
- `src/views/manage`
- `src/views/home`

- `src/components`
- `src/hooks`

- `src/theme`
- `src/styles`
- `packages`

## 七、关键认知

学习后台管理系统时，要建立 5 个关键认知。

### 认知 1：后台项目的主线是业务数据流转

一个典型流程是：

1. 用户登录。
2. 前端保存 token。
3. 请求用户信息。
4. 根据角色生成可访问路由。
5. 根据路由生成菜单。
6. 用户点击菜单进入页面。
7. 页面请求列表数据。
8. 用户搜索、分页、新增、编辑、删除。

后续课程基本都会围绕这条主线展开。

### 认知 2：路由不是只负责跳转

在后台项目中，路由还经常负责：

- 页面权限。
- 菜单标题。
- 菜单图标。
- 菜单排序。
- 面包屑。
- 页面缓存。
- 外链跳转。

所以学后台项目时，路由是重点。

### 认知 3：请求封装决定项目稳定性

后台系统接口多、错误多、权限状态复杂。请求封装如果混乱，页面代码会很难维护。

好的请求封装应该让页面只关心业务数据，而不是反复处理 token、错误 code 和异常提示。

### 认知 4：CRUD 是后台业务开发的基本功

大多数后台需求都可以拆成：

- 查列表。
- 搜索。
- 新增。
- 编辑。
- 删除。
- 看详情。
- 改状态。

所以能不能高质量完成 CRUD 页面，是判断是否掌握后台开发的关键。

### 认知 5：不要一开始就钻进封装

这个项目封装较多，初学时不要先看 `packages` 或所有 hooks。

更好的顺序是：

1. 先看页面功能。
2. 再看页面调用了哪些 hooks。
3. 再看 hooks 封装了哪些重复逻辑。
4. 最后再研究公共包。
