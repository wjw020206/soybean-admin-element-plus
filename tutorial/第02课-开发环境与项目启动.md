# 第 2 课：开发环境与项目启动

## 课程定位

第 1 课建立了后台管理系统的整体地图，第 2 课开始真正进入项目。

本课的目标不是深入业务代码，而是把项目在本地稳定跑起来，并理解启动一个现代 Vue 后台项目时，Node、pnpm、workspace、环境变量和 npm scripts 分别扮演什么角色。

很多初学者遇到后台项目时，第一道坎不是 Vue 语法，而是环境问题：Node 版本不对、包管理器用错、依赖安装不完整、monorepo workspace 没理解、启动命令和构建命令混用。解决这些问题后，后续阅读源码和做业务功能才有稳定基础。

## 学习目标

学完本课后，你应该能够：

- 正确理解本项目的 Node 和 pnpm 版本要求。
- 知道为什么本项目推荐使用 pnpm，而不是 npm 或 yarn。
- 理解 `pnpm-workspace.yaml` 在 monorepo 项目中的作用。
- 能够说明 `pnpm dev`、`pnpm build`、`pnpm preview` 的区别。
- 能够从启动日志中观察端口、运行模式、代理和访问地址。
- 能够定位项目启动失败时常见问题的排查方向。

## 建议课时

建议用 60-90 分钟完成本课：

- 15 分钟：检查 Node 和 pnpm 环境。
- 15 分钟：阅读 `package.json` 和 workspace 配置。
- 20 分钟：安装依赖并启动项目。
- 20 分钟：理解启动命令、构建命令和预览命令。
- 10 分钟：记录启动日志并完成自检。

## 一、为什么开发环境很重要

现代前端项目不是只有源码，还依赖一整套工具链：

- Node.js 负责运行构建工具、脚本和开发服务器。
- pnpm 负责安装依赖、解析 workspace 包和执行 scripts。
- Vite 负责开发服务器、热更新和打包构建。
- TypeScript 负责类型检查。
- ESLint 负责代码规范检查。
- Git hooks 负责提交前校验。

如果环境不一致，同一份代码可能在别人机器上能跑，在你机器上报错。

后台管理项目通常依赖更多工具和内部包，因此环境稳定性更重要。本项目使用 pnpm monorepo 管理，根项目依赖了多个 `packages/*` 下的 workspace 包，如果包管理器使用不正确，很容易出现依赖解析失败。

## 二、本项目的环境要求

先看 `package.json` 中的 engines 配置：

```json
{
  "engines": {
    "node": ">=20.19.0",
    "pnpm": ">=8.7.0"
  }
}
```

这表示项目期望：

- Node.js 版本不低于 `20.19.0`。
- pnpm 版本不低于 `8.7.0`。

再看 `README.md` 中的环境准备说明：

- NodeJS：`>=20.19.0`，推荐 `20.19.0` 或更高。
- pnpm：`>=10.5.0`，推荐 `10.5.0` 或更高。

这里可以这样理解：

- `package.json` 是项目工具层面的最低约束。
- `README.md` 给的是作者推荐的使用版本。
- 实际学习时，建议优先使用 README 推荐版本，也就是 Node `20.19.0+` 和 pnpm `10.5.0+`。

## 三、检查本机环境

在项目根目录执行：

```bash
node -v
pnpm -v
```

你应该看到类似：

```bash
v20.19.0
10.5.0
```

不要求版本号完全一致，但要满足最低版本要求。

如果 Node 版本过低，可能遇到：

- Vite 启动失败。
- 依赖安装失败。
- TypeScript 或 ESLint 工具运行异常。
- 某些 ESM 模块解析失败。

如果 pnpm 版本过低，可能遇到：

- workspace 依赖解析异常。
- lockfile 不兼容。
- 安装依赖时报版本或协议错误。

## 四、为什么不要用 npm 或 yarn

本项目 README 明确说明：

```bash
pnpm i
```

并强调：

> 由于本项目采用了 pnpm monorepo 的管理方式，因此请不要使用 npm 或 yarn 来安装依赖。

原因主要有三个。

### 1. 本项目使用 pnpm workspace

根目录有 `pnpm-workspace.yaml`：

```yaml
packages:
  - "packages/*"
```

这表示 `packages` 目录下的每个子包都是当前 workspace 的一部分。

例如 `package.json` 中有这些依赖：

```json
{
  "@sa/alova": "workspace:*",
  "@sa/axios": "workspace:*",
  "@sa/color": "workspace:*",
  "@sa/hooks": "workspace:*",
  "@sa/materials": "workspace:*",
  "@sa/utils": "workspace:*"
}
```

`workspace:*` 的意思是：这些包不是从 npm 仓库下载的，而是从当前项目的 `packages/*` 目录中解析。

### 2. pnpm-lock.yaml 由 pnpm 维护

本项目根目录有 `pnpm-lock.yaml`。

lockfile 的作用是锁定依赖版本，保证不同开发者安装出来的依赖尽可能一致。

如果你用 npm 或 yarn 安装，可能会生成：

- `package-lock.json`
- `yarn.lock`

这会造成依赖管理混乱，也可能让 workspace 包无法按项目预期解析。

### 3. pnpm 的依赖结构更适合 monorepo

pnpm 使用内容寻址存储和符号链接机制，可以更高效地管理多个包之间的依赖。

对于 monorepo 项目，这种方式有几个好处：

- 避免重复安装大量依赖。
- workspace 包之间引用更清晰。
- 安装速度更快。
- 依赖版本更可控。

所以在这个项目里，记住一条规则：安装依赖、启动项目、构建项目，都使用 pnpm。

## 五、理解 package.json scripts

`package.json` 中最重要的是 `scripts`：

```json
{
  "scripts": {
    "build": "vite build --mode prod",
    "build:test": "vite build --mode test",
    "cleanup": "sa cleanup",
    "commit": "sa git-commit",
    "commit:zh": "sa git-commit -l=zh-cn",
    "dev": "vite --mode test",
    "dev:prod": "vite --mode prod",
    "gen-route": "sa gen-route",
    "lint": "eslint . --fix",
    "postbuild": "sa print-soybean",
    "prepare": "simple-git-hooks",
    "preview": "vite preview",
    "release": "sa release",
    "typecheck": "vue-tsc --noEmit --skipLibCheck",
    "update-pkg": "sa update-pkg"
  }
}
```

本课先关注 5 个命令。

### 1. `pnpm dev`

```bash
pnpm dev
```

实际执行：

```bash
vite --mode test
```

含义：

- 启动 Vite 开发服务器。
- 使用 `test` 模式。
- 会读取 `.env` 和 `.env.test`。
- 支持热更新。
- 适合日常开发。

### 2. `pnpm dev:prod`

```bash
pnpm dev:prod
```

实际执行：

```bash
vite --mode prod
```

含义：

- 启动开发服务器。
- 但使用 `prod` 模式环境变量。
- 会读取 `.env` 和 `.env.prod`。
- 适合本地调试生产环境配置。

注意：它仍然是开发服务器，不等于生产构建。

### 3. `pnpm build`

```bash
pnpm build
```

实际执行：

```bash
vite build --mode prod
```

含义：

- 执行生产构建。
- 使用 `prod` 模式。
- 输出静态资源。
- 用于部署前打包。

### 4. `pnpm build:test`

```bash
pnpm build:test
```

实际执行：

```bash
vite build --mode test
```

含义：

- 执行测试环境构建。
- 使用 `.env.test`。
- 适合部署到测试环境。

### 5. `pnpm preview`

```bash
pnpm preview
```

实际执行：

```bash
vite preview
```

含义：

- 预览已经构建出来的产物。
- 通常要先执行 `pnpm build`。
- 不是开发服务器。
- 不适合日常开发。

## 六、安装依赖

第一次拉取项目后，在项目根目录执行：

```bash
pnpm i
```

这个命令会做几件事：

- 读取 `package.json`。
- 读取 `pnpm-lock.yaml`。
- 读取 `pnpm-workspace.yaml`。
- 安装根项目依赖。
- 解析 `packages/*` 中的 workspace 包。
- 建立本地依赖链接。
- 执行必要的生命周期脚本。

安装完成后，根目录通常会出现或更新：

- `node_modules`
- `.pnpm`
- workspace 包链接

如果已经存在 `node_modules`，一般不需要重复安装，除非依赖发生变化。

## 七、启动项目

在项目根目录执行：

```bash
pnpm dev
```

根据 `package.json`，它会启动：

```bash
vite --mode test
```

再结合 `vite.config.ts`：

```ts
server: {
  host: '0.0.0.0',
  port: 9527,
  open: true,
  proxy: createViteProxy(viteEnv, enableProxy)
}
```

可以得出：

- 本地开发服务器端口是 `9527`。
- `host: '0.0.0.0'` 表示允许局域网访问。
- `open: true` 表示启动后自动打开浏览器。
- 开发服务器会配置代理。

启动成功后，通常可以访问：

```txt
http://localhost:9527/
```

如果终端显示了 Network 地址，也可以用局域网 IP 访问，例如：

```txt
http://192.168.x.x:9527/
```

## 八、理解 mode 和环境变量

本项目有这些环境文件：

- `.env`
- `.env.test`
- `.env.prod`

Vite 的规则是：

- 公共配置放在 `.env`。
- 测试模式配置放在 `.env.test`。
- 生产模式配置放在 `.env.prod`。

当执行：

```bash
pnpm dev
```

实际模式是 `test`，所以会读取：

- `.env`
- `.env.test`

当执行：

```bash
pnpm dev:prod
```

实际模式是 `prod`，所以会读取：

- `.env`
- `.env.prod`

当执行：

```bash
pnpm build
```

实际模式也是 `prod`，所以会读取：

- `.env`
- `.env.prod`

本项目 `.env.test` 中配置了：

```bash
VITE_SERVICE_BASE_URL=https://mock.apifox.cn/m1/3109515-0-default
```

这表示 `pnpm dev` 默认连接测试环境 mock 接口。

`.env.prod` 中也配置了：

```bash
VITE_SERVICE_BASE_URL=https://mock.apifox.cn/m1/3109515-0-default
```

当前仓库的测试和生产接口地址都指向 mock 地址，后续接入真实后端时，通常会修改这里。

## 九、启动日志应该观察什么

启动项目时，不要只看“能不能打开页面”，还要学会看终端日志。

重点观察这些信息。

### 1. 当前运行模式

因为 `pnpm dev` 对应：

```bash
vite --mode test
```

所以你要知道当前读取的是 `.env.test`。

这会影响：

- 接口地址。
- 代理配置。
- 是否启用某些环境能力。

### 2. 访问地址

重点看：

- Local 地址。
- Network 地址。
- 端口是否为 `9527`。

如果端口被占用，Vite 可能会提示换端口，也可能启动失败，具体取决于配置和运行情况。

### 3. 代理信息

`.env` 中有：

```bash
VITE_HTTP_PROXY=Y
VITE_PROXY_LOG=Y
```

这表示开发环境会启用 HTTP 代理，并显示代理日志。

代理配置来自：

- `vite.config.ts`
- `build/config`
- `.env`
- `.env.test`

后续学习接口请求时，代理会很重要。

### 4. 构建或编译错误

如果启动失败，常见错误包括：

- 依赖缺失。
- Node 版本过低。
- pnpm 版本不兼容。
- 端口占用。
- TypeScript 类型生成异常。
- workspace 包解析失败。
- 环境变量格式错误。

遇到错误时，不要只复制最后一行，要从第一条 error 开始看。

## 十、`pnpm dev`、`pnpm build`、`pnpm preview` 的区别

这三个命令非常容易混淆。

### `pnpm dev`

用于开发。

特点：

- 启动 Vite dev server。
- 有热更新。
- 修改源码后浏览器自动更新。
- 不会生成最终部署产物。
- 默认读取 `.env` 和 `.env.test`。

适合：

- 写页面。
- 调试接口。
- 看效果。
- 学习源码。

### `pnpm build`

用于构建。

特点：

- 生成静态资源。
- 会做打包压缩。
- 没有热更新。
- 默认读取 `.env` 和 `.env.prod`。
- 产物用于部署。

适合：

- 部署前打包。
- 检查生产构建是否通过。
- 发现开发环境不暴露的构建问题。

### `pnpm preview`

用于预览构建产物。

特点：

- 不是开发模式。
- 通常需要先执行 `pnpm build`。
- 用本地服务器查看打包后的页面。
- 端口在 `vite.config.ts` 中配置为 `9725`。

适合：

- 构建后本地验收。
- 检查静态资源路径。
- 检查生产模式页面是否能正常打开。

简单记忆：

```txt
dev     = 开发时用
build   = 部署前打包
preview = 看打包后的结果
```

## 十一、为什么 preview 之前通常要先 build

`pnpm preview` 预览的是构建产物。如果你没有先构建，可能没有可预览的最新产物。

推荐流程：

```bash
pnpm build
pnpm preview
```

然后访问：

```txt
http://localhost:9725/
```

这里的 `9725` 来自 `vite.config.ts`：

```ts
preview: {
  port: 9725
}
```

## 十二、常见启动问题和排查方向

### 1. Node 版本过低

现象：

- 安装依赖失败。
- 启动 Vite 失败。
- 报 Node engine 不满足。

排查：

```bash
node -v
```

解决：

- 切换到 Node `20.19.0+`。
- 推荐使用 nvm、fnm 或 volta 管理 Node 版本。

### 2. pnpm 版本不合适

现象：

- workspace 解析失败。
- lockfile 报错。
- `workspace:*` 依赖无法识别。

排查：

```bash
pnpm -v
```

解决：

- 升级到 pnpm `10.5.0+`。

### 3. 用错包管理器

现象：

- 出现 `package-lock.json` 或 `yarn.lock`。
- 本地依赖结构和项目预期不一致。
- workspace 包解析异常。

解决：

- 使用 `pnpm i`。
- 不要混用 npm、yarn 和 pnpm。

### 4. 端口被占用

现象：

- `9527` 无法启动。
- 终端提示端口占用。

排查：

- 看终端提示。
- 检查是否已有一个 dev server 正在运行。

解决：

- 关闭占用端口的进程。
- 或临时修改 `vite.config.ts` 中的 `server.port`。

### 5. 接口请求失败

现象：

- 页面能打开，但登录或接口报错。
- Network 中请求失败。

排查：

- 查看 `.env.test` 中的 `VITE_SERVICE_BASE_URL`。
- 查看 `.env` 中的 `VITE_HTTP_PROXY`。
- 查看终端代理日志。
- 查看浏览器 Network 面板。

解决：

- 确认接口地址可访问。
- 确认代理配置正确。
- 后续课程会专门讲请求封装和代理。

## 十五、关键认知

### 认知 1：启动命令不是随便起的

`pnpm dev` 并不是 pnpm 自己提供的固定命令，而是执行了 `package.json` 中的 `scripts.dev`。

也就是说：

```bash
pnpm dev
```

等价于：

```bash
pnpm run dev
```

再等价于：

```bash
vite --mode test
```

理解这一点后，你以后看到任何项目，都应该先看 `package.json` 的 scripts。

### 认知 2：mode 决定环境变量

本项目 `dev` 使用的是 `test` mode，而不是默认 development。

这意味着开发时读取的接口地址来自 `.env.test`。

如果你调接口时发现地址不对，第一反应应该是检查当前命令使用了哪个 mode。

### 认知 3：monorepo 项目不能随便换包管理器

本项目使用 pnpm workspace，很多依赖来自本地 `packages`。

如果你用 npm 或 yarn 安装依赖，很可能破坏依赖结构。

学习这个项目时，统一使用 pnpm。

### 认知 4：开发服务器和构建产物不是一回事

`pnpm dev` 是开发时用的服务器，有热更新。

`pnpm build` 是打包，用于生成部署产物。

`pnpm preview` 是预览构建产物，不是用来写代码的。

### 认知 5：先让项目稳定运行，再深入源码

如果项目没有稳定跑起来，直接读业务代码会非常低效。

正确顺序是：

1. 检查环境。
2. 安装依赖。
3. 启动项目。
4. 观察页面。
5. 观察日志。
6. 再进入源码阅读。

执行：

```bash
node -v
pnpm -v
```

记录结果：

```md
- Node.js 版本：
- pnpm 版本：
- 是否满足项目要求：
```

### 作业 2：记录 scripts 含义

阅读 `package.json`，整理：

```md
# scripts 说明

- pnpm dev：
- pnpm dev:prod：
- pnpm build：
- pnpm build:test：
- pnpm preview：
- pnpm typecheck：
- pnpm lint：
```

要求不要只复制命令，要写出每个命令的用途。

### 作业 4：区分 dev、build、preview

用自己的话解释：

```md
# dev、build、preview 的区别

## pnpm dev

## pnpm build

## pnpm preview
```

要求：

- 必须说明是否有热更新。
- 必须说明是否生成部署产物。
- 必须说明分别适合什么场景。

## 十七、自检问题
