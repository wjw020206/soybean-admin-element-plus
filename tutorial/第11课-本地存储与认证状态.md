# 第 11 课：本地存储与认证状态

## 课程定位

第 10 课我们已经把登录流程跑通了，知道了：

```txt
点击登录
  -> 调登录接口
  -> 拿到 token / refreshToken
  -> 写入本地存储
  -> 获取用户信息
  -> 跳转页面
```

但还有一个问题必须继续往下追：

```txt
这些登录状态到底存在哪里？
页面刷新后为什么还能保持登录？
退出登录时为什么不仅要清 token，还要清 tabs 和路由状态？
```

这就是本课要解决的问题。

后台管理系统里的“认证状态”不是只有一个 token。真实项目里通常至少包括：

- token
- refreshToken
- 当前用户信息
- 上一个登录用户 id
- 已缓存的 tabs
- 主题和布局相关的持久化配置

这些状态一部分存在 Pinia 里，一部分存在浏览器存储里。只有把“内存状态”和“持久化状态”一起看，你才能真正理解后台项目的登录会话。

## 二、先建立整体认识：认证状态分成两层

这个项目里的认证状态分成两层：

### 1. 内存层

主要在 Pinia store 里，例如：

```ts
const token = ref(getToken());

const userInfo: Api.Auth.UserInfo = reactive({
  userId: '',
  userName: '',
  roles: [],
  buttons: []
});
```

这些状态的特点是：

- 响应式。
- 页面中直接可用。
- 刷新页面后会丢失，需要重新初始化。

### 2. 持久化层

主要在 `localStorage` 中，例如：

- `token`
- `refreshToken`
- `lastLoginUserId`
- `globalTabs`

这些状态的特点是：

- 浏览器刷新后仍然存在。
- 需要手动读取和写入。
- 不具备 Vue 响应式能力。

所以你可以把登录态理解成：

```txt
Pinia 负责运行中的状态
localStorage 负责刷新后的恢复依据
```

## 三、`src/utils/storage.ts`：项目自己的存储入口

当前文件内容很短：

```ts

const storagePrefix = import.meta.env.VITE_STORAGE_PREFIX || '';
```

这说明项目不是在业务代码里直接到处写：

```ts

window.localStorage.getItem(...)
```

而是统一从这里拿封装好的对象：

- `localStg`
- `sessionStg`
- `localforage`

其中当前项目真正广泛使用的是：

```txt
localStg
```

## 四、`localStg` 和 `sessionStg` 有什么区别

看这一行：

```ts
export const localStg = createStorage<StorageType.Local>('local', storagePrefix);
```

和：

```ts
export const sessionStg = createStorage<StorageType.Session>('session', storagePrefix);
```

区别在第一个参数：

```txt
'local'   -> window.localStorage
'session' -> window.sessionStorage
```

也就是说：

- `localStg` 底层用 `localStorage`
- `sessionStg` 底层用 `sessionStorage`

它们的浏览器行为不同：

### `localStorage`

- 刷新页面还在
- 关闭浏览器后通常还在
- 更适合 token、主题设置、语言、tabs 缓存

### `sessionStorage`

- 刷新页面还在
- 关闭当前标签页 / 浏览器会话后清掉
- 更适合生命周期更短的临时状态

但要注意：**这个项目当前认证相关状态主要都放在 `localStorage`，不是 `sessionStorage`。**

## 五、真正的存储封装在 `packages/utils/src/storage.ts`

`src/utils/storage.ts` 只是入口，真正实现是在：

[packages/utils/src/storage.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/packages/utils/src/storage.ts)

核心代码：

```ts
export function createStorage<T extends object>(type: StorageType, storagePrefix: string) {
  const stg = type === 'session' ? window.sessionStorage : window.localStorage;

  const storage = {
    set<K extends keyof T>(key: K, value: T[K]) {
      const json = JSON.stringify(value);

      stg.setItem(`${storagePrefix}${key as string}`, json);
    },

    get<K extends keyof T>(key: K): T[K] | null {
      const json = stg.getItem(`${storagePrefix}${key as string}`);
      if (json) {
        let storageData: T[K] | null = null;

        try {
          storageData = JSON.parse(json);
        } catch {}

        if (storageData) {
          return storageData as T[K];
        }
      }

      stg.removeItem(`${storagePrefix}${key as string}`);

      return null;
    },

    remove(key: keyof T) {
      stg.removeItem(`${storagePrefix}${key as string}`);
    },

    clear() {
      stg.clear();
    }
  };

  return storage;
}
```

这个封装很值得学，因为它很典型。

## 六、为什么要自己封装 `localStorage`

直接用原生 API 当然也可以：

```ts
localStorage.setItem('token', token);
const token = localStorage.getItem('token');
```

但项目还是封了一层，主要是为了四件事。

### 1. 统一 JSON 序列化 / 反序列化

原生 `localStorage` 只能存字符串。

所以像：

```ts
localStg.set('globalTabs', tabs.value);
```

这种数组、对象都必须先 `JSON.stringify`。

封装后业务层就不用每次手写：

```ts
JSON.stringify(...)
JSON.parse(...)
```

### 2. 统一加前缀

比如：

```ts
SOY_token
SOY_refreshToken
SOY_globalTabs
```

统一前缀能避免不同系统、不同环境下 key 冲突。

### 3. 增加类型约束

通过泛型：

```ts
createStorage<StorageType.Local>(...)
```

可以让：

```ts
localStg.set('token', 'abc')
```

合法，而：

```ts
localStg.set('token', true)
```

报类型错误。

### 4. 统一存储接口

业务代码里只记：

```ts
set
get
remove
clear
```

不需要每次关心底层是 `localStorage` 还是 `sessionStorage`。

## 七、`storagePrefix` 是什么

在 [src/utils/storage.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/utils/storage.ts) 中：

```ts
const storagePrefix = import.meta.env.VITE_STORAGE_PREFIX || '';
```

而 `.env` 中配置了：

```env
VITE_STORAGE_PREFIX=SOY_
```

所以最终存到浏览器里的 key 实际上不是：

```txt
token
refreshToken
globalTabs
```

而是：

```txt
SOY_token
SOY_refreshToken
SOY_globalTabs
```

## 八、为什么要加存储前缀

这通常是为了避免冲突。

想象这种场景：

- 你同一个域名下部署了多个前端系统。
- 你本地开发时不同项目都跑在相似环境里。
- 浏览器里残留了旧项目的 token。

如果大家都直接写：

```txt
token
lang
themeSettings
```

就很容易互相污染。

加上前缀后：

```txt
SOY_token
SOY_lang
SOY_themeSettings
```

至少这套项目自己的存储 key 更容易隔离。

`.env` 里这条注释写得很明确：

```env
# Used to differentiate storage across different domains
VITE_STORAGE_PREFIX=SOY_
```

虽然注释写的是 “across different domains”，但更准确地说，它的核心价值是：

```txt
区分不同应用的存储命名空间
```

## 十、认证主线里最重要的两个存储项：`token` 和 `refreshToken`

登录成功后，在 `auth store` 中：

```ts
localStg.set('token', loginToken.token);
localStg.set('refreshToken', loginToken.refreshToken);
```

这两个值是认证状态恢复和请求鉴权的核心依据。

### `token`

它通常用于每次请求时放到请求头：

```txt
Authorization: Bearer xxx
```

### `refreshToken`

它用于 token 过期时刷新认证状态。

本课不深入刷新机制细节，第 13 课会专门讲。但你现在至少要知道：

```txt
token 是当前工作凭证
refreshToken 是续期凭证
```

## 十一、读取 token 的入口非常统一

在 [src/store/modules/auth/shared.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/auth/shared.ts)：

```ts
export function getToken() {
  return localStg.get('token') || '';
}
```

注意项目没有在每个地方都直接写：

```ts
localStg.get('token')
```

而是抽成：

```ts
getToken()
```

这样做的好处是：

- token 读取入口统一。
- 以后 token 的读取逻辑变化时，改一处就行。
- 认证相关逻辑更集中在 `auth/shared.ts`。

你可以把它理解为：

```txt
storage 是底层工具
auth/shared.ts 是认证领域里的存储门面
```

## 十二、为什么刷新页面后还能保持登录

看 `auth store` 初始化：

```ts
const token = ref(getToken());
```

这意味着当 store 初始化时，会先去 `localStorage` 读取 token。

如果浏览器里已经有：

```txt
SOY_token
```

那 `token.value` 一开始就不是空字符串。

但这还不够，因为 `userInfo` 仍然是空的。

所以还有：

```ts
async function initUserInfo() {
  const hasToken = getToken();

  if (hasToken) {
    const pass = await getUserInfo();

    if (!pass) {
      resetStore();
    }
  }
}
```

也就是说，刷新后的恢复过程是：

```txt
浏览器里还保留 token
  -> auth store 初始化时读到 token
  -> 再请求 getUserInfo()
  -> 恢复 userInfo / roles / buttons
```

所以真正的刷新恢复不是“把所有状态都存一份”，而是：

```txt
持久化最关键的认证凭证
再用它重新拉取用户上下文
```

这是一种更合理的做法。

## 十三、退出登录时为什么不只清 token

很多初学者会觉得退出登录就是：

```ts
localStg.remove('token');
localStg.remove('refreshToken');
```

但这个项目的 `resetStore()` 明显做得更多：

```ts
async function resetStore() {
  recordUserId();

  clearAuthStorage();

  authStore.$reset();

  if (!route.meta.constant) {
    await toLogin();
  }

  tabStore.cacheTabs();
  routeStore.resetStore();
}
```

它做了五类事情：

### 1. 记录上一个用户 id

```ts
recordUserId();
```

### 2. 清理认证存储

```ts
clearAuthStorage();
```

### 3. 重置 auth store 内存状态

```ts
authStore.$reset();
```

### 4. 必要时跳转登录页

```ts
await toLogin();
```

### 5. 处理 tabs 和路由状态

```ts
tabStore.cacheTabs();
routeStore.resetStore();
```

这说明：

```txt
退出登录不是单纯清 token，
而是把整套“已登录上下文”都恢复到未登录状态。
```

## 十四、`clearAuthStorage()` 为什么只清两项

它的实现非常简单：

```ts
export function clearAuthStorage() {
  localStg.remove('token');
  localStg.remove('refreshToken');
}
```

注意它只清：

- `token`
- `refreshToken`

不清：

- `lastLoginUserId`
- `globalTabs`
- `lang`
- `themeSettings`

这是因为这些值职责不同。

### 为什么不清 `lastLoginUserId`

因为它要留给“下次登录时判断是不是切换用户”使用。

### 为什么不清 `globalTabs`

因为有的退出场景希望先缓存 tabs，再由后续逻辑决定是否恢复或清空。

### 为什么不清主题和语言

因为这些是应用偏好，不应因为退出登录就丢掉。

这体现了一个很重要的工程思路：

```txt
不要用一个“大清空”把所有东西一刀切
而是按状态职责分别处理
```

## 十五、`lastLoginUserId` 是用来解决什么问题的

在 `auth store` 中：

```ts
function recordUserId() {
  if (!userInfo.userId) {
    return;
  }

  localStg.set('lastLoginUserId', userInfo.userId);
}
```

以及：

```ts
function checkTabClear(): boolean {
  if (!userInfo.userId) {
    return false;
  }

  const lastLoginUserId = localStg.get('lastLoginUserId');

  if (lastLoginUserId !== userInfo.userId) {
    localStg.remove('globalTabs');
    tabStore.clearTabs();

    return true;
  }

  return false;
}
```

这套逻辑要解决的是：

```txt
上一次登录的是用户 A，
这一次登录的是用户 B，
那就不应该继续保留 A 的 tabs 缓存
```

为什么？

因为 tabs 里可能包含：

- A 有权限看到、B 没权限看到的页面
- A 上次打开过、B 完全不相关的页面
- 带 query / cache 的用户上下文

所以用户切换时要清 tabs，是很合理的。

而 `lastLoginUserId` 就是用来判断：

```txt
当前登录用户和上一次登录用户是不是同一个人
```

## 十六、`globalTabs` 是什么时候写入的

在 `tab store` 中：

```ts
function cacheTabs() {
  if (!themeStore.tab.cache) return;

  localStg.set('globalTabs', tabs.value);
}
```

并且页面关闭或刷新前会触发：

```ts
useEventListener(window, 'beforeunload', () => {
  cacheTabs();
});
```

这表示：

```txt
如果开启了 tabs 缓存配置，
当前打开的 tabs 会在页面刷新 / 关闭前写入 localStorage
```

然后初始化 tab store 时：

```ts
const storageTabs = localStg.get('globalTabs');

if (themeStore.tab.cache && storageTabs) {
  const extractedTabs = extractTabsByAllRoutes(router, storageTabs);
  tabs.value = updateTabsByI18nKey(extractedTabs);
}
```

这说明 tabs 的恢复也是基于 `localStorage` 完成的。

## 十七、为什么用户切换时要 `remove('globalTabs')`

看这句：

```ts
localStg.remove('globalTabs');
tabStore.clearTabs();
```

这里同时做了两件事：

### 1. 清持久化缓存

```ts
localStg.remove('globalTabs');
```

防止下次刷新页面时又把旧 tabs 恢复回来。

### 2. 清当前内存状态

```ts
tabStore.clearTabs();
```

让当前页面运行时立刻生效。

这又体现了“内存层 + 持久化层”要同步处理的原则。

## 十八、存储中的认证状态和路由行为是联动的

路由守卫里会用：

```ts
const isLogin = Boolean(localStg.get('token'));
```

这说明浏览器存储里的 token 直接影响路由行为。

例如：

### 有 token

```txt
路由守卫认为你处于已登录状态
继续尝试初始化用户信息和权限路由
```

### 没 token

```txt
路由守卫认为你未登录
访问受保护页面时跳登录页
```

所以如果你手动删掉浏览器里的：

```txt
SOY_token
```

再访问需要登录的页面，路由行为就会变化。

这也是本课作业里让你手动删 token 的原因。

## 十九、`get()` 为什么会在解析失败后顺手删除数据

`createStorage.get()` 的实现里有：

```ts
const json = stg.getItem(`${storagePrefix}${key as string}`);
if (json) {
  let storageData: T[K] | null = null;

  try {
    storageData = JSON.parse(json);
  } catch {}

  if (storageData) {
    return storageData as T[K];
  }
}

stg.removeItem(`${storagePrefix}${key as string}`);

return null;
```

意思是：

```txt
如果 key 对应的值存在，但 JSON 解析失败，
或者解析后的结果不符合预期，
那干脆把这条脏数据删掉，再返回 null。
```

这样做的好处是：

- 防止坏数据一直污染后续逻辑。
- 下一次读取时不会反复出错。

这是一个很实用的“容错式存储读取”思路。

不过你也要注意，这里有一个实现细节：

```ts
if (storageData) {
  return storageData as T[K];
}
```

它把 `false`、`0`、`''` 这类合法值也当成“空值”了。这是实现上的一个取舍，后面你自己搭项目时可以思考要不要改成更严格的判断：

```ts
if (storageData !== null)
```

本课先重点理解设计思路，不急着改。

## 二十、`createLocalforage()` 是什么，为什么当前项目几乎没用

在：

```ts
export const localforage = createLocalforage<StorageType.Local>('local');
```

你会看到 `localforage` 也被导出了。

它底层基于：

```txt
localforage
```

是一个异步存储工具，可以切换 driver：

- `local`
- `indexedDB`
- `webSQL`

但当前仓库里几乎没有实际业务在用它，主要在用的是：

```txt
localStg
```

原因很现实：

- 认证状态和轻量配置更适合简单直接的 `localStorage`
- 同步 API 用起来更方便
- token、语言、主题、tabs 都不是大体积数据

所以你现在可以把 `createLocalforage()` 理解成：

```txt
工具库预留的更复杂持久化能力
```

但本项目当前主线认证逻辑几乎不依赖它。

## 二十一、这套存储设计的工程价值

把本课内容总结一下，这套设计有几个明显优点。

### 1. 存储入口统一

所有业务代码都通过：

```ts
localStg
sessionStg
```

访问存储，而不是自己散写原生 API。

### 2. 状态职责分清楚

- 认证凭证存 `token / refreshToken`
- 用户切换判断存 `lastLoginUserId`
- tabs 缓存存 `globalTabs`
- 主题偏好存 `themeSettings / darkMode / themeColor`

### 3. 内存与持久化联动明确

退出登录、切换用户、刷新恢复，都会同时考虑：

- store 内存状态
- localStorage 持久化状态

### 4. 类型约束更稳

`StorageType.Local` 让每个 key 的类型更清晰，降低误写概率。
