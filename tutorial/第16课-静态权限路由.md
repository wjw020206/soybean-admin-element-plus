# 第 16 课：静态权限路由

## 课程定位

第 15 课我们已经把路由守卫和访问控制顺序看清楚了，知道了：

```txt
先初始化路由
  -> 再判断是否登录
  -> 再判断是否有角色权限
  -> 最后分流到正常页面、403 或 404
```

但还有一个关键问题没有彻底讲透：

```txt
这些“权限路由”本身到底从哪里来？
为什么有的路由是 constant，有的路由是 auth？
静态模式下角色限制是写在哪里的？
generated routes 和 custom routes 到底怎么拼起来？
```

这一课就专门讲这条链路。

这也是你以后自己从零搭后台项目时，最常见的第一版权限方案：

```txt
前端自己维护路由表
前端自己维护角色字段
登录后根据当前用户 roles 过滤出可访问路由
```

这就是“静态权限路由”。

## 二、先明确：什么叫“静态权限路由”

在这个项目里，权限模式由 `.env` 决定：

```env
VITE_AUTH_ROUTE_MODE=static
```

它表示：

```txt
路由结构和角色权限关系，主要由前端本地代码维护。
```

也就是说：

- 前端自己定义哪些页面存在
- 前端自己定义页面的 `meta.roles`
- 登录后根据当前用户的 `roles` 过滤出可访问路由

这和动态权限路由的区别非常直接：

### 静态模式

```txt
路由定义在前端代码里
角色过滤也在前端做
```

### 动态模式

```txt
后端返回当前用户可访问路由
前端负责转换和注册
```

所以这一课的核心不是“怎么请求路由接口”，而是：

```txt
前端本地这一整套路由定义、拆分、过滤、注册流程是怎么跑起来的。
```

## 三、静态模式的路由源头有两份

看 [src/router/routes/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/routes/index.ts)：

```ts
import { generatedRoutes } from '../elegant/routes';
```

然后文件里还自己定义了一份：

```ts
const customRoutes: CustomRoute[] = [...]
```

所以静态模式下，路由源头实际上有两部分：

### 1. `generatedRoutes`

来自：

[src/router/elegant/routes.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/elegant/routes.ts)

它是由 elegant-router 插件根据约定生成的路由描述。

例如你会看到：

```ts
{
  name: 'home',
  path: '/home',
  component: 'layout.base$view.home',
  meta: {
    title: 'home',
    i18nKey: 'route.home',
    icon: 'mdi:monitor-dashboard',
    order: 1
  }
}
```

以及：

```ts
{
  name: 'function_super-page',
  path: '/function/super-page',
  component: 'view.function_super-page',
  meta: {
    title: 'function_super-page',
    i18nKey: 'route.function_super-page',
    icon: 'ic:round-supervisor-account',
    order: 5,
    roles: ['R_SUPER']
  }
}
```

这说明：

```txt
页面、路径、布局、meta 信息可以通过约定式方式生成。
```

### 2. `customRoutes`

这是作者手写补充的路由。

例如：

- `/exception`
- `/document`
- `document_project-link` 这种带 `meta.href` 的外链页面

这些场景往往不适合完全靠约定式目录自动生成，或者作者就是希望集中手写定义。

所以：

```txt
generatedRoutes 解决“按目录自动生成”
customRoutes 解决“手工补充和特殊路由”
```

## 四、为什么要同时保留 generated 和 custom 两套来源

因为后台项目里路由不是纯粹的一种风格。

### 只用 generatedRoutes 的问题

如果全靠目录自动生成，像下面这些场景会不够灵活：

- 外链菜单
- 一些特殊嵌套路由
- 需要集中管理的一组演示页
- 不完全符合目录即路由的页面结构

### 只用 customRoutes 的问题

如果全部手写，维护成本又会升高：

- 新增页面要手写路由
- 文件路径和路由配置容易脱节
- 大量基础页面重复劳动多

所以这个项目选的是折中方案：

```txt
常规页面尽量由插件生成
特殊页面和补充页面由 customRoutes 手写
最后再统一合并。
```

这个设计很适合中后台项目。

## 五、`createStaticRoutes()` 是静态模式的总入口

看 [src/router/routes/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/router/routes/index.ts)：

```ts
export function createStaticRoutes() {
  const constantRoutes: ElegantRoute[] = [];

  const authRoutes: ElegantRoute[] = [];

  [...customRoutes, ...generatedRoutes].forEach(item => {
    if (item.meta?.constant) {
      constantRoutes.push(item);
    } else {
      authRoutes.push(item);
    }
  });

  return {
    constantRoutes,
    authRoutes
  };
}
```

这个函数非常关键。

它做的事情其实很简单：

```txt
先把 customRoutes 和 generatedRoutes 合并成一份总路由表，
再按照 meta.constant 拆成 constantRoutes 和 authRoutes 两组。
```

这里要注意一个事实：

```txt
constant 和 auth 的划分，不是靠文件位置，不是靠目录名，
而是靠 route.meta.constant。
```

## 六、constant routes 和 auth routes 的区别

### 1. constant routes

含义是：

```txt
不依赖当前用户角色，也通常不需要登录就能访问，或者至少它是系统的固定基础路由。
```

典型例子在 generated routes 里有：

```ts
{
  name: '403',
  path: '/403',
  meta: {
    constant: true,
    hideInMenu: true
  }
}
```

还有：

```ts
{
  name: '404',
  path: '/404',
  meta: {
    constant: true,
    hideInMenu: true
  }
}
```

以及：

```ts
{
  name: 'login',
  path: '/login/:module(...)',
  meta: {
    constant: true,
    hideInMenu: true
  }
}
```

这些页面显然不能等“用户角色过滤后再决定有没有”。

### 2. auth routes

含义是：

```txt
这些路由会进入权限体系，
后续可能根据角色过滤掉。
```

比如：

```ts
{
  name: 'function_super-page',
  ...
  meta: {
    roles: ['R_SUPER']
  }
}
```

它就是典型的 auth route。

## 七、静态模式下，角色权限到底写在哪里

答案很明确：

```txt
写在路由的 meta.roles 里。
```

例如：

```ts
meta: {
  title: 'function_super-page',
  roles: ['R_SUPER']
}
```

这表示：

```txt
只有拥有 R_SUPER 角色的用户，才允许访问这个路由。
```

如果 `roles` 为空或根本没写，就表示：

```txt
这个页面进入权限体系，但不细分角色限制。
```

注意区分这两个层次：

### 没有 `meta.constant`

表示：

```txt
这条路由属于 auth route，要进入权限路由初始化流程。
```

### `meta.roles` 为空

表示：

```txt
它虽然是 auth route，但不限制具体角色。
```

这两个不是一回事。

## 八、静态模式下，超级角色怎么放行

看 [src/store/modules/auth/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/auth/index.ts)：

```ts
const isStaticSuper = computed(() => {
  const { VITE_AUTH_ROUTE_MODE, VITE_STATIC_SUPER_ROLE } = import.meta.env;

  return VITE_AUTH_ROUTE_MODE === 'static' && userInfo.roles.includes(VITE_STATIC_SUPER_ROLE);
});
```

而 `.env` 中是：

```env
VITE_STATIC_SUPER_ROLE=R_SUPER
```

也就是说：

```txt
在静态权限模式下，如果当前用户 roles 里包含 R_SUPER，
那他会被视为超级角色。
```

后面初始化静态权限路由时就会直接放行全部 auth routes。

## 九、静态权限路由是怎么初始化的

看 [src/store/modules/route/index.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/route/index.ts)：

```ts
async function initAuthRoute() {
  if (!authStore.userInfo.userId) {
    await authStore.initUserInfo();
  }

  if (authRouteMode.value === 'static') {
    initStaticAuthRoute();
  } else {
    await initDynamicAuthRoute();
  }

  tabStore.initHomeTab();
}
```

静态模式走的是：

```ts
initStaticAuthRoute();
```

继续看：

```ts
function initStaticAuthRoute() {
  const { authRoutes: staticAuthRoutes } = createStaticRoutes();

  if (authStore.isStaticSuper) {
    addAuthRoutes(staticAuthRoutes);
  } else {
    const filteredAuthRoutes = filterAuthRoutesByRoles(staticAuthRoutes, authStore.userInfo.roles);

    addAuthRoutes(filteredAuthRoutes);
  }

  handleConstantAndAuthRoutes();

  setIsInitAuthRoute(true);
}
```

这一段就是静态权限路由的核心执行链。

主线可以写成：

```txt
createStaticRoutes()
  -> 拿到本地 authRoutes
  -> 如果是超级角色，直接全量加入
  -> 否则按 userInfo.roles 过滤
  -> addAuthRoutes(...)
  -> handleConstantAndAuthRoutes()
```

## 十、`filterAuthRoutesByRoles()` 是怎么过滤的

看 [src/store/modules/route/shared.ts](/Users/codepencil/Documents/project/个人/soybean-admin-element-plus/src/store/modules/route/shared.ts)：

```ts
export function filterAuthRoutesByRoles(routes: ElegantConstRoute[], roles: string[]) {
  return routes.flatMap(route => filterAuthRouteByRoles(route, roles));
}
```

核心在递归函数：

```ts
function filterAuthRouteByRoles(route: ElegantConstRoute, roles: string[]): ElegantConstRoute[] {
  const routeRoles = (route.meta && route.meta.roles) || [];

  const isEmptyRoles = !routeRoles.length;
  const hasPermission = routeRoles.some(role => roles.includes(role));

  const filterRoute = { ...route };

  if (filterRoute.children?.length) {
    filterRoute.children = filterRoute.children.flatMap(item => filterAuthRouteByRoles(item, roles));
  }

  if (filterRoute.children?.length === 0) {
    return [];
  }

  return hasPermission || isEmptyRoles ? [filterRoute] : [];
}
```

你要重点看懂它的四条规则。

### 规则 1：当前路由没写 `roles`

那就默认允许：

```ts
const isEmptyRoles = !routeRoles.length;
```

### 规则 2：当前路由写了 `roles`

只要用户命中其中一个角色，就允许：

```ts
const hasPermission = routeRoles.some(role => roles.includes(role));
```

### 规则 3：对子路由递归过滤

```ts
filterRoute.children = filterRoute.children.flatMap(item => filterAuthRouteByRoles(item, roles));
```

这说明：

```txt
权限过滤不是只看第一层菜单，而是整棵路由树递归处理。
```

### 规则 4：如果父路由过滤后没有任何子路由，就把这个父路由也删掉

```ts
if (filterRoute.children?.length === 0) {
  return [];
}
```

这条规则很重要。

因为如果一个父菜单下面所有子页面都没权限了，那么这个父菜单自己再保留着也没有意义。

## 十一、过滤后的路由接下来怎么变成真正可用路由

过滤只是第一步。真正让页面能访问、菜单能显示，还要走后续流程。

在 `initStaticAuthRoute()` 里：

```ts
handleConstantAndAuthRoutes();
```

继续看这个方法：

```ts
function handleConstantAndAuthRoutes() {
  const allRoutes = [...constantRoutes.value, ...authRoutes.value];

  const sortRoutes = sortRoutesByOrder(allRoutes);

  const vueRoutes = getAuthVueRoutes(sortRoutes);

  resetVueRoutes();

  addRoutesToVueRouter(vueRoutes);

  getGlobalMenus(sortRoutes);

  getCacheRoutes(vueRoutes);
}
```

这表示静态权限路由后面还要继续经过这几个步骤：

```txt
constant + auth 合并
  -> 按 meta.order 排序
  -> 转成 Vue Router routes
  -> 动态 addRoute 到 router
  -> 生成全局菜单
  -> 提取 keepAlive 缓存页
```

也就是说，权限过滤不是终点，它只是路由运行态构建的一环。

## 十三、静态权限路由为什么适合开发阶段

代码里有一段注释：

```ts
It recommends to use static mode in the development environment...
```

这是有道理的。

静态模式的优点很明显：

### 1. 前端开发效率高

不用等后端接口就能把菜单、页面、权限结构先跑起来。

### 2. 调试简单

角色权限全在本地代码里，看一眼 `meta.roles` 就知道怎么回事。

### 3. 新增页面成本低

你只要：

- 新建页面文件
- 让插件生成路由，或手写 custom route
- 配好 `meta`

就能直接看到效果。

### 4. 学习成本更低

对你现在这种“以读源码和自己搭后台项目为目标”的阶段，静态模式非常适合入门。

## 十四、静态模式的局限也要知道

静态模式不是没有代价。

它的问题主要有：

### 1. 权限规则写死在前端

如果后端权限经常变化，前端也要跟着改代码。

### 2. 菜单和路由配置无法完全由后台管理

运营或管理员不能只改数据库就调整路由结构。

### 3. 角色体系一复杂，前端维护成本会上升

尤其是：

- 角色很多
- 路由树很深
- 菜单经常调整

这种情况下，动态模式通常更合适。

所以静态模式不是“更高级”，而是：

```txt
更适合开发期、演示项目、权限结构相对稳定的系统。
```

## 十八、本课小结

这一课你应该已经建立起下面这条主线：

```txt
静态权限模式
  -> 路由来源 = customRoutes + generatedRoutes
  -> 按 meta.constant 拆成 constantRoutes 和 authRoutes
  -> authRoutes 按当前用户 roles 递归过滤
  -> constant + auth 合并后排序
  -> 转成 Vue Router 路由并动态注册
  -> 再生成菜单、缓存页等运行态数据
```

你真正要记住的是这句话：

```txt
静态权限路由不是“写几个 roles 字段”这么简单，
而是一套“本地路由定义 -> 权限过滤 -> Router/Menu 运行态生成”的完整流程。
```
